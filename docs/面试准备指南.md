# FeedLens 面试准备指南

> **项目**：基于 LangGraph + DeepSeek 的多 Agent 智能信息简报系统  
> **版本**：v2.2.0 | **日期**：2026-06-25

---

## 一、面试介绍模板

### 1.1 一句话描述（15 秒）

> 我做的是一个基于 LangGraph（Agent 编排框架）和 DeepSeek（大语言模型）的多 Agent 智能信息简报系统。它能定时从 RSS 源采集新闻，经过向量去重、多因子排序、LLM 生成简报后推送给用户，并通过用户反馈持续学习偏好。核心亮点是主 Agent 通过 ReAct 循环（推理-行动循环）自主编排三个子 Agent，正常流程全部规则化路由，零 LLM 开销。

### 1.2 两分钟电梯演讲

> FeedLens 解决的是信息过载问题——用户每天面对数十个信息源，手动筛选耗时且低效。我设计了一个多 Agent 系统：
>
> - **主 Agent** 使用 LangGraph StateGraph（状态图）编排流程，8 个节点通过 ReAct 循环（推理-行动循环）自主决策。路由策略是规则优先+LLM 兜底，正常流程 0 次 LLM 路由调用。
> - **三个子 Agent**：Collection Agent（采集 Agent，RSS 采集+搜索补充，支持 Pipeline/ReAct 双模式）、Ranking Agent（排序 Agent，三层向量去重+四因子加权排序）、Briefing Agent（简报 Agent，结构化 JSON 生成+代码层质量审查）。
> - **个性化**：用户反馈通过 EMA（指数移动平均，α=0.3）平滑更新偏好向量，排序权重在冷启动（反馈<3 条）和暖启动间动态切换。
> - **性能优化**：经过 10 项系统化优化，单次执行从 8 分钟降到 2 分钟以内，LLM API 调用从 35+ 次降到 6-8 次。
>
> 技术栈包括 LangGraph、DeepSeek Chat、ChromaDB（向量数据库）、SQLite、bge-small-zh-v1.5（本地 Embedding 模型）、APScheduler（定时调度）、Streamlit（UI 框架）和 MCP 协议（模型上下文协议）。

### 1.3 五分钟深度介绍

在 2 分钟版本基础上补充：

> **架构演进**方面，最初是单 Agent 线性 DAG（有向无环图），后来重构为多 Agent 自主规划。关键变化是引入了规则优先路由、向量预过滤（prefilter，跨批次去重）、批量 LLM 裁决、Pipeline/ReAct 双模式。
>
> **数据层**使用 SQLite（11 张表，WAL 模式，即预写式日志）+ ChromaDB（3 个 Collection，即向量集合）混合存储。结构化数据走 SQL，语义向量走 ChromaDB，设计哲学是"能用 SQL 解决的不用向量，能用规则判断的不用 LLM"。
>
> **容错设计**包括：子 Agent 执行隔离（`run_with_isolation`，单个失败不阻塞管线）、三层 JSON 解析降级、Planner（编排器）死循环检测、模型回退链（`LLMRouter` 顺序尝试多个 Provider）、执行栅栏（`execution_fence`，per-user 锁防并发）。
>
> **13 个技术决策**（ADR，即 Architecture Decision Record，架构决策记录）覆盖了 Agent 框架选型、去重策略、排序权重、Embedding 选型、存储方案、部署方案等关键决策点。

## 二、面试高频问题及回答要点

### Q1：为什么选择 LangGraph 而不是 LangChain Agent？

**回答要点**：
- LangChain Agent 是黑盒决策，状态不可控、不可观测
- LangGraph 提供显式的 StateGraph（状态图），每个节点、每条边都是显式定义的，可追踪、可调试
- StateGraph 的 TypedDict State（类型化字典状态）在节点间共享，天然支持多 Agent 协作
- 支持条件边（conditional edges，根据状态动态决定下一节点）实现 LLM 动态路由，也支持固定边做规则路由
- 内置 checkpointer（检查点持久化）支持状态持久化（虽然 MVP 暂未开启）

**参考文档**：[ADR-001](../技术决策记录/ADR-001-Agent编排框架选型.md)

---

### Q2：去重策略为什么用"向量 + LLM"两阶段，而不是纯向量或纯 LLM？

**回答要点**：
- **纯向量的问题**：标题不同但内容相同的文章（如不同媒体转述同一事件），向量相似度可能落在模糊区间
- **纯 LLM 的问题**：每条都送 LLM 判断成本太高，100 条就是 4950 对比较
- **两阶段策略**：
  - 余弦相似度 ≥ 0.88 → 直接判重（节省 LLM 调用）
  - ≤ 0.70 → 直接保留
  - 0.70~0.88 → 收集所有待裁决 pair，**一次性批量送 LLM**（而非逐对调用），HTTP 往返减少 80-95%
- 超限保护：待裁决 pair > 20 对时，超限部分按 0.80 硬判重

**参考文档**：[ADR-004](../技术决策记录/ADR-004-向量去重策略.md)

---

### Q3：排序的冷启动和偏好模式是怎么切换的？

**回答要点**：
- **冷启动**（feedback_count（反馈数量）< 3）：similarity（相似度）0.40 + recency（时效性）0.25 + preference（偏好）0.10 + importance（重要性）0.25
  - 以当次 Goal（用户目标）匹配度为主，偏好权重仅 10%
- **暖启动**（feedback_count ≥ 3）：similarity 0.30 + recency 0.20 + **preference 0.40** + importance 0.10
  - 偏好权重提升到 40%，减少 Goal 相似度依赖
- 偏好向量通过 EMA（指数移动平均，α=0.3）平滑更新，正负向量分离（v_like（喜欢向量）/ v_dislike（不喜欢向量））
- 设计了 feedback_bias（反馈偏置）临时补偿项，EMA 更新后自然归零，避免单次反馈永久污染

**参考文档**：[ADR-005](../技术决策记录/ADR-005-排序权重动态切换.md)

---

### Q4：为什么要用规则优先路由而不是全 LLM 路由？

**回答要点**：
- **问题**：v2.0 阶段全 LLM 路由，每个节点间跳转都调用 LLM，正常流程产生 6-10 次额外 API 调用
- **分析**：正常流程 7 个路由场景都是确定性的（plan（执行计划）非空→执行、执行完→评估、通过→推送...），根本不需要 LLM
- **方案**：`_rule_based_router_decision()`（基于规则的路由决策函数）覆盖所有正常流程，仅在 `needs_retry`（需要重试）或 `overall_pass=false`（整体审查未通过）等异常场景才升级到 LLM
- **效果**：正常流程路由 LLM 调用 -100%，延迟降低 60%+
- **设计哲学**：LLM 只做"创造"（生成简报、编排决策），不做"判断"（路由跳转、质量评分）

**参考文档**：[main_agent.md](../项目各模块的设计思路/main_agent.md)

---

### Q5：为什么用 bge-small-zh-v1.5 而不是 OpenAI Embedding？

**回答要点**：
- **成本**：本地运行，零 API 费用。每天采集 100+ 条目，向量化频率高
- **延迟**：本地推理 < 10ms/条，API 调用 > 100ms/条（含网络往返）
- **隐私**：用户偏好向量不离开本地
- **效果**：bge-small-zh-v1.5 在中文语义相似度任务上 C-MTEB 排名靠前，384 维向量足够本场景使用
- **代价**：模型约 130MB，首次运行自动下载

**参考文档**：[ADR-006](../技术决策记录/ADR-006-Embedding模型选型.md)

---

### Q6：为什么用 SQLite + ChromaDB 混合存储，而不是单一数据库？

**回答要点**：
- **SQLite 擅长**：结构化查询（按时间、来源、状态过滤）、事务保证、关系 JOIN
- **ChromaDB 擅长**：向量相似度检索（SQLite 不支持）、语义去重、偏好匹配
- **分工**：
  - SQLite 存全量业务数据（11 张表）：用户、信息源、条目、简报、反馈、执行日志
  - ChromaDB 存 3 个向量集合：条目历史（跨批次预过滤）、用户偏好、领域知识
- **设计哲学**：能用 SQL 解决的不用向量，能用向量解决的不用 LLM

**参考文档**：[ADR-007](../技术决策记录/ADR-007-数据存储选型.md)

---

### Q7：简报质量检查为什么不用 LLM 自评，而是代码层评分？

**回答要点**：
- **LLM 自评的问题**：LLM 倾向于给自己的输出打高分（self-enhancement bias），不可靠；且增加 API 调用
- **代码层方案**：四维评分公式
  - completeness（完整性）：预期 categories（分类）覆盖了多少
  - coherence（连贯性）：结构化检查（必填字段、格式合规）
  - relevance（相关性）：与 Goal（用户目标）的语义匹配度（向量相似度，非 LLM）
  - conciseness（简洁性）：摘要长度控制
- 质量 ≥ 0.7 直接完成，否则自动重试（最多 2 次）
- **设计哲学**：LLM 做"创造"（生成），代码做"判断"（评分）

**参考文档**：[briefing_agent.md](../项目各模块的设计思路/briefing_agent.md)

---

### Q8：用户反馈怎么影响排序结果？

**回答要点**：
- **三级反馈**：like（喜欢，+1）、dislike（不喜欢，-1）、irrelevant（不相关，-2，权重更高）
- **EMA 平滑更新**（指数移动平均）：`v_new = α × v_feedback + (1-α) × v_old`，α=0.3
- **正负向量分离**：v_like（喜欢向量）和 v_dislike（不喜欢向量）分别维护，排序时取差值
- **feedback_bias**（反馈偏置）临时补偿：like +0.15, dislike -0.10, irrelevant -0.15，EMA 更新后归零
- **异步处理**：反馈记录不阻塞主流程，通过 FeedbackAgent（反馈 Agent）异步更新偏好向量
- **冷启动保护**：feedback_count（反馈数量）< 3 时偏好权重仅 10%，防止少量反馈过度影响

**参考文档**：[ADR-009](../技术决策记录/ADR-009-用户反馈机制.md)

### Q9：性能从 8 分钟优化到 2 分钟，具体做了什么？

**回答要点**（10 项优化）：

| # | 优化项 | 节省 |
|---|--------|------|
| 01 | enrich_metadata（元数据增强）批量+可关闭 | 关闭时省 13 次 API |
| 02 | observe（观察节点）结果判断优化 | 修复误判，避免无效重跑 |
| 03 | briefing（简报）质量检查+重试 | completeness（完整性）分母修正 |
| 04 | Planner/Router 规则化降级 | 正常流程省 6 次 router LLM |
| 05 | collection pipeline（采集流水线）固定化 | 采集 LLM 调用 -100% |
| 06 | thinking_mode（思考模式）关闭+tool_choice（工具选择） | 修复 function calling（函数调用）不稳定 |
| 07 | 向量预过滤（prefilter）跨批次去重 | 进入 Ranking 条目 -70%+ |
| 08 | briefing 管线深度优化 | ReAct 思考 -100%，重试 3→2 |
| 09 | 去重算法批量 LLM 裁决 | HTTP 往返 -80~95% |
| 10 | 摘要清洗+Planner 容错+VS（向量存储）单例 | 三层清洗+三层 JSON 解析 |

**核心原则**：LLM 只做"创造"，不做"判断"。

**参考文档**：[FeedLens_项目总结.md](../FeedLens_项目总结.md) §六

### Q10：如果让你重新设计，会做哪些不同的选择？

**回答要点**（展示反思能力）：
1. **多用户支持**：MVP 阶段就应设计用户隔离（独立 Collection + 独立 SQLite 视图），现在改造代价大
2. **子 Agent 并行化**：Collection 和 Ranking 之间可以部分并行（采集完一批就开始排序，不等全部采集完）
3. **流式输出**：简报生成应支持 SSE 流式，用户可以看到生成过程，体验更好
4. **测试覆盖**：目前是脚本式测试而非 pytest，缺少单元测试和 CI
5. **配置热更新**：config.yaml 修改后需重启，应支持文件监听自动 reload

## 三、技术深挖问题

### T1：LangGraph 的 StateGraph 和 MessageGraph 有什么区别？为什么选 StateGraph？

- MessageGraph（消息图）以消息列表为核心，适合对话场景
- StateGraph（状态图）以自定义 TypedDict（类型化字典）为核心，适合多 Agent 协作场景
- FeedLens 需要 39 个字段的复杂状态（编排控制、子 Agent 结果、质量审查、路由控制），StateGraph 更合适
- StateGraph 支持 reducer 函数（归并函数，如 append（追加）列表），多个节点可安全并发写入

### T2：MCP（Model Context Protocol，模型上下文协议）在项目中怎么用的？

- **Search Server**（搜索服务，SSE :8100，即 Server-Sent Events 服务端推送）：Collection Agent 在 RSS 采集不足时调用，搜索补充内容
- **Push Server**（推送服务，stdio，即标准输入输出）：主 Agent 推送简报时调用，写入 JSONL 通知队列
- 选择 FastMCP 框架：Python 原生，轻量，支持 SSE（长连接）和 stdio（一次性调用）两种传输模式
- 设计考量：搜索走 SSE（长连接，适合多次交互），推送走 stdio（一次性调用，无需保持连接）

### T3：ReAct 循环（推理-行动循环）的终止条件是什么？

1. **硬上限**：`agentic_turn_count`（Agent 轮次计数）≥ `max_turns`（最大轮次，默认 5，config.yaml 可配）
2. **死循环检测**：连续 3 次相同路由决策 → 强制收敛
3. **正常完成**：push_notification（推送通知）→ update_memory（更新记忆）→ END
4. **子 Agent 隔离**：单个子 Agent 失败不终止，设置 `agent_status`（Agent 状态）为 `isolated`（已隔离）后继续

### T4：跨批次向量预过滤（prefilter）是怎么工作的？

- Collection Agent 采集到新条目后，立即对每条计算标题 embedding（向量嵌入）
- 查询 ChromaDB `feed_items`（条目向量集合）Collection，余弦相似度 ≥ 0.92 的判定为已见过
- 纯标题编码（与写入端一致），不经过 LLM
- 效果：进入 Ranking 的条目数减少 70%+

### T5：模型的回退链（fallback chain）是怎么设计的？

- `LLMRouter`（LLM 路由器）封装，按配置的 `llm.fallback`（LLM 回退列表）列表顺序尝试
- 主 Provider（模型提供商，DeepSeek）不可用时自动切换到备用 Provider
- 每个 Provider 独立配置 api_key（API 密钥）、base_url（基础 URL）、model（模型名称）
- 第一个成功的响应即返回，不等待其他 Provider

## 四、行为面试问题

### B1：项目中遇到的最大技术挑战是什么？

> 最大的挑战是性能优化。v2.0 初版单次执行需要 4 分种左右，35+ 次 LLM API 调用，其中 65% 是浪费的。我通过日志分析定位了 10 个瓶颈点，系统化地逐一优化。关键洞察是"LLM 只做创造，不做判断"——把路由决策、质量评分、预过滤（prefilter）这些可规则化的判断从 LLM 移到代码层，最终把执行时间降到 2 分钟以内，API 调用降到 6-8 次。

### B2：你是怎么做技术选型的？

> 每个关键决策都写了 ADR（Architecture Decision Record，架构决策记录），记录了上下文、备选方案、决策理由和后果。比如 Agent 框架选了 LangGraph 而不是 LangChain Agent，因为需要显式的状态图（StateGraph）来控制多 Agent 协作流程；Embedding 选了本地 bge-small-zh-v1.5 而不是 OpenAI API，因为每天需要大量向量化操作，本地推理零成本、低延迟。

### B3：如果项目要上线生产环境，还需要做什么？

> 1. 多用户认证和隔离
> 2. Docker 容器化部署
> 3. Redis 分布式锁替代 per-user Lock（当前是 threading.Lock 单进程锁）
> 4. 数据库迁移到 PostgreSQL（并发写入支持更好）
> 5. 监控和告警（Prometheus + Grafana）
> 6. CI/CD 流水线（持续集成/持续部署）
> 7. 压力测试和性能基线

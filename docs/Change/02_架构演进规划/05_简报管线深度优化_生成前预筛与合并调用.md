# 05 架构演进 — 简报管线深度优化：生成前预检 + 消除 ReAct + 降低重试

> 基于 2026-06-24 全项目代码分析及可行性论证，从架构层面设计简报管线的深度优化方案。
> **核心思路**：取消不可靠的 LLM 自评，通过「生成前硬编码预检 + 分离 coherence + quality 仅评 relevance + 降低重试上限」减少无效 LLM 调用。

---

## 一、问题诊断

### 1.1 架构层面的根本问题

FeedLens 简报管线的核心问题是 **LLM 过度参与确定性流程**。具体表现为：

```
当前架构:
  ranked_items → ReAct LLM思考 → generate_briefing(LLM) → ReAct LLM思考 → quality_check(LLM)
                  ↑ 不必要的决策         ↑ 必要的生成        ↑ 不必要的决策    ↑ 可优化

  问题：
  1. ReAct LLM 在"生成→检查→重试"这个确定性循环中每轮都要"思考"下一步做什么
  2. quality_check 中 coherence 的三条规则（URL/时间/重要性）都是硬编码可检测的
  3. completeness 可在生成前预判（条目数已知），重试无法改善
  4. 重试上限 3 次偏高，多数场景第 3 次重试已是边际改善

目标架构:
  ranked_items → _preflight_for_briefing (硬编码预检, 0次LLM)
      ↓
  generate_briefing (LLM) → 自动评估 (硬编码, quality LLM仅首次)
      ↓ 质量不达标 → 自动重试(最多2次) → 强制收敛
```

### 1.2 简报被退回重生成的根本原因

质量评分公式：`score = completeness×0.3 + relevance×0.4 + coherence×0.3`

| 因子 | 权重 | 是否可预判 | 优化方向 |
|------|------|-----------|---------|
| completeness | 0.3 | ✅ 条目数已知 | 生成前预判，避免无效重试 |
| relevance | 0.4 | ⚠️ 需 LLM 评估 | 保留独立 LLM，但仅首次调用（缓存复用） |
| coherence | 0.3 | ✅ 规则可检测 | 移到生成前阶段（URL去重+时间跨度警告） |

## 二、架构设计原则

### 2.1 核心原则

> **LLM 只做「创造」不做「判断」**

| 任务类型 | 是否适合 LLM | 理由 |
|---------|------------|------|
| 生成简报 JSON（摘要、标题、分类） | ✅ 适合 | 需要语言理解和创意 |
| 评估 relevance（与目标相关度） | ⚠️ 需要 | 需要语义理解，但可缓存复用 |
| 决定下一步操作（generate vs finish） | ❌ 不适合 | 流程是确定的，规则即可 |
| completeness 计算 | ❌ 不适合 | 纯数学计算 |
| URL/时间 矛盾检测 | ❌ 不适合 | 规则匹配即可 |
| **LLM 自评质量** | ❌ **不可靠** | 利益冲突，LLM 倾向于给自己打高分 |

### 2.2 设计决策

| 决策 | 理由 |
|------|------|
| **取消 LLM 自评** | LLM 自评存在利益冲突，倾向于打高分（偏差 ~0.05-0.10） |
| **生成前硬编码预检** | URL去重、时间跨度、条目数量都可在生成前处理，消除可控扣分 |
| **分离 coherence 到生成前** | 三条规则（URL/时间/重要性）不需要 LLM |
| **quality LLM 缓存复用** | relevance 仅首次调用，后续重试复用 |
| **ReAct 思考消除** | "生成→检查→重试"是确定性的，代码层直接控制 |
| **降低重试上限** | 3→2 次，因可控扣分因素已消除 |

## 三、架构变更详情

### 3.1 Briefing Agent 内部流程重构

**Before**:
```
run_briefing_agent() ReAct 循环:
  Turn 1: LLM思考 → generate_briefing(LLM)
  Turn 2: LLM思考 → quality_check(LLM)
  Turn 3: (if quality<0.7) LLM思考 → generate...
  ...
```

**After**:
```
run_briefing_agent() 确定性流程:
  ranked_items → _preflight_for_briefing() (硬编码预检: URL去重+时间跨度+数量)
       ↓
  generate_briefing (LLM) → brief_quality_check_node (硬编码+quality LLM仅首次)
       ↓
  quality >= 0.7? → 自动完成
  quality < 0.7 且 count < 3? → 自动重试 (注入反馈)
  count >= 3? → 强制收敛
```

### 3.2 工具注册表变更

**Before**: briefing 阶段工具 = `[generate_briefing, quality_check, finish_task]`

**After**: briefing 阶段工具 = `[generate_briefing, finish_task]`

`quality_check` 保留函数实现但不再暴露给 LLM（phase 改为 `briefing_legacy`），改为代码层直接调用 `_llm_assess_quality`。

### 3.3 数据流变更

**Before**:
```
generate_briefing → briefing (JSON)
    ↓
quality_check (LLM) → brief_quality (score)
    ↓
ReAct LLM 决定: finish_task or retry
```

**After**:
```
_preflight_for_briefing (硬编码) → 预处理后的条目 + 警告
    ↓
generate_briefing (LLM) → briefing (JSON)
    ↓
硬编码后校验 (quality LLM仅首次, 后续复用缓存) → brief_quality (score)
    ↓
硬编码决定: quality >= 0.7 → finish | else → retry (最多2次)
```

## 五、实施路线图

### Phase 1: 预检 + generate_node 修改（低风险）
- 新增 `_preflight_for_briefing()` 函数
- 修改 `generate_briefing_node()` 调用预检并注入警告到 prompt
- **不改** ReAct 循环和 quality_check

### Phase 2: quality_check 适配（中风险）
- 修改 `brief_quality_check_node()`：coherence 规则移到预检，relevance 缓存复用
- quality LLM 仅首次调用

### Phase 3: ReAct 简化（需要充分测试）
- 修改 `BRIEFING_SYSTEM_PROMPT` 移除 quality_check
- 修改 `run_briefing_agent()` 增加硬编码自动评估+自动重试
- 移除 ReAct LLM 思考步骤，重试上限 3→2

### Phase 4: 清理
- `tool_registry` 中 quality_check phase 改为 briefing_legacy
- 性能测试与对比

## 六、风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| relevance 缓存复用偏差 | 低 | relevance 是整体平均分，重试主要调整结构而非内容 |
| 预检 URL 去重过于激进 | 低 | 保留 importance 更高的条目 |
| ReAct 消除后异常场景 | 低 | generate_count 硬限制 + max_turns 双重兜底 |
| 重试上限降低（3→2） | 低 | 可控扣分因素已消除，实际需要重试的场景大幅减少 |

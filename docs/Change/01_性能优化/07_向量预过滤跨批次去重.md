# 07 P1 — 向量预过滤：跨批次去重（Collection Agent 内部步骤）

> 在 Collection Agent 内部，每次采集到条目后通过 ChromaDB 向量查重拦截历史重复条目，用 ~2s 的 CPU 向量计算避免将重复条目送入昂贵的 LLM 处理管线。
> **预过滤是 Collection Agent 的内部实现细节，Planner 无需感知。**
>
> **状态**：✅ 已实施（2026-06-25，commit `6be31d6`），实际实施与原始规划有两处关键优化。

---

## 一、问题分析

### 1.1 当前去重的边界限制

`tools/fc_tools.py:335` 的 `deduplicate()` 函数是一个**批次内去重**系统：

```
62 条采集数据
  │
  ▼
deduplicate():
  ├── bge-small 生成 embedding (62 次)
  ├── O(62²) = ~1900 次余弦比对
  ├── sim >= 0.88 → 直接判重
  ├── 0.70 ≤ sim < 0.88 → LLM 裁决（最多 20 次 API 调用）
  └── 返回 unique_items
```

**它不检查这些条目是否在上周、上个月的执行中已经采集过。** RSS 源每天推送的新闻有大量交叉重复——同一事件多源报道、昨日已推送的持续跟踪报道、定期更新的数据报告等。这些重复条目每次都完整走一遍 Collection → Ranking → Briefing 的 LLM 管线。

### 1.2 浪费量化（以 62 条为例）

| 阶段 | 处理条目 | LLM 调用次数 | 耗时占比 |
|------|---------|-------------|---------|
| Collection Agent (RSS) | 62 条 | 0（固定流程） | ~5s |
| Ranking Agent → dedup | 62 条 | 0-20（LLM 裁决中间区） | ~10s |
| Ranking Agent → rank_items | 62 条 | 1（LLM 整体打分） | ~60s |
| Briefing Agent | Top 10 | 1-3 | ~20s |
| **合计** | | | **~95s** |

假设 62 条中 **45 条是历史重复**（保守估计），则：
- 45 × (rank_items 评分) ≈ **43 秒 LLM 时间白白浪费**
- 45 × (O(n²) embedding 比对) ≈ 额外的 CPU 开销
- 去重中 LLM 裁决的 20 次上限也可能被历史重复消耗

### 1.3 已有基础设施

项目已经有向量预过滤需要的全部核心组件：

| 组件 | 文件 | 状态 |
|------|------|------|
| bge-small-zh-v1.5 embedding 模型 | `utils/embedding.py` | ✅ 已加载（6.73ms/条） |
| ChromaDB 持久化客户端 | `models/vector_store.py` | ✅ `feed_items` 集合已定义 |
| 向量相似度检索 | `models/vector_store.py:115` `search_similar()` | ✅ 可直接用 |
| 条目历史数据 | SQLite `raw_items` + `deduped_items` | ✅ 30 天保留 |
| 去重阈值配置 | `config.yaml:39-43` `ranking` 段 | ✅ 可复用 |

**唯独缺失的一环**：在 Collection Agent 内部，采集条目后查 ChromaDB 是否有相似历史条目。

## 二、方案设计

### 2.1 核心思路

预过滤是 **Collection Agent 的内部步骤**，不是 Planner 编排的独立节点：

- Collection Agent 对上层承诺"返回新信息"
- `fetch_rss` / `search_web` 拿到条目后，Collection Agent 自己查向量库去重
- 只将**新条目**（历史未见过）返回给上层
- Planner 完全不需要知道这个机制的存在

```
Collection Agent 内部流程:
  fetch_rss → 62 条原始数据
       │
       ▼
  ┌─────────────────────────────────┐
  │ _prefilter_against_history()     │  ← Collection Agent 内部函数
  │                                 │
  │  for each item:                 │
  │    embedding = bge(title+summary)│
  │    result = ChromaDB.query(     │
  │      embedding,                 │
  │      collection="feed_items",   │
  │      threshold=0.92             │
  │    )                            │
  │    if result has close match:   │
  │      → discard (历史已有)       │
  │    else:                        │
  │      → keep (新数据)            │
  │                                 │
  │  62 条 → ~17 条新数据            │
  └─────────────────────────────────┘
       │
       ▼
  normalize_items → 返回 collected_items (仅新条目)
```

### 2.2 阈值策略

| 相似度区间 | 处理策略 | 理由 |
|-----------|---------|------|
| ≥ 0.92 | **直接丢弃** | 标题+摘要高度重合，99% 是同一篇文章的转载或轻微更新 |
| < 0.92 | **保留**，交给 Ranking Agent 的 dedup 做精细 LLM 裁决 | 同主题不同角度、续报、更新等内容仍需 LLM 判断 |

阈值选 0.92 而非 0.88（去重高阈值）的原因：
- 预过滤是**拦截层**，目标是"明显重复"，不是"可能重复"
- 过高（0.95+）→ 漏网太多，预过滤形同虚设
- 过低（0.85 以下）→ 可能误杀同主题不同角度的新内容，且这类判断正是 LLM 擅长的
- 0.92 在 BGE 384 维空间中对"同标题+同摘要"场景有很高的区分力，同时保留"同事件不同报道"给 LLM

### 2.6 新增节点独立性与降级

```python
def _vector_prefilter(state: dict) -> dict:
    """节点独立性：任何异常都降级为透传全部条目，不阻塞管线。"""
    items = state.get("collected_items", [])
    if not items:
        return {}

    try:
        vs = _get_vector_store()
        vs.init_collections()
        em = EmbeddingModel()

        threshold = 0.92  # 从 config 读取
        kept, discarded = [], []

        for item in items:
            text = f"{item.get('title', '')} {item.get('summary', '')}"
            vec = em.encode_single(text)
            results = vs.search_similar(
                query_embedding=vec,
                n_results=1,
                collection="feed_items",
            )
            # ChromaDB 返回 distance，需转换为 cosine similarity
            # 如果余弦相似度 >= threshold，视为重复
            dist = results.get("distances", [[1.0]])[0]
            if dist and len(dist) > 0:
                cos_sim = 1 - dist[0]  # ChromaDB 默认用 L2/余弦 距离
                if cos_sim >= threshold:
                    discarded.append(item)
                    continue
            kept.append(item)

        print(f"[vector_prefilter] 预过滤: {len(items)}→{len(kept)}条 "
              f"(丢弃{len(discarded)}条历史重复)", flush=True)
        return {
            "collected_items": kept,
            "prefilter_discarded_count": len(discarded),
        }
    except Exception as e:
        print(f"[vector_prefilter] 降级透传: {e}", flush=True)
        return {}  # 不修改 state，全部条目继续走原流程
```

### 改动 1 详细：`_prefilter_against_history` 函数（Collection Agent 内部）

```python
# agents/collection_agent.py 新增函数（在 run_collection_pipeline 之前）
def _prefilter_against_history(items: list, threshold: float = 0.92, top_k: int = 1) -> tuple:
    """向量预过滤：查询 ChromaDB 历史条目，丢弃明显重复项（无 LLM）。

    Collection Agent 内部函数，对上层透明。

    Returns:
        (kept_items, discarded_items): 保留的条目和丢弃的条目
    """
    if not items:
        return items, []

    try:
        from models.vector_store import VectorStore
        from utils.embedding import EmbeddingModel

        vs = VectorStore(persist_dir="data/chroma")
        vs.init_collections()
        em = EmbeddingModel()

        kept, discarded = [], []

        for item in items:
            title = item.get("title", "")
            summary = item.get("summary", "")
            text = f"{title} {summary}".strip()
            if not text:
                kept.append(item)
                continue

            try:
                vec = em.encode_single(text)
                results = vs.search_by_embedding(
                    vec.tolist() if hasattr(vec, 'tolist') else vec,
                    n_results=top_k,
                    collection="feed_items",
                )
                distances = results.get("distances", [[]])
                if distances and distances[0]:
                    cos_sim = 1.0 - distances[0][0]
                    if cos_sim >= threshold:
                        discarded.append(item)
                        continue
                kept.append(item)
            except Exception:
                kept.append(item)  # 单条异常不阻塞整体

        drop_rate = len(discarded) / len(items) * 100 if items else 0
        print(
            f"[collection_prefilter] {len(items)}→{len(kept)}条 "
            f"(丢弃{len(discarded)}条, {drop_rate:.0f}%历史重复)",
            flush=True,
        )
        return kept, discarded
    except Exception as e:
        print(f"[collection_prefilter] 降级透传全部{len(items)}条: {e}", flush=True)
        return items, []


# 在 run_collection_pipeline() 中的调用方式：
# Step 1: fetch_rss 后
rss_result = tool_registry.dispatch("fetch_rss", {"sources": sources})
rss_items = rss_result.get("items", [])
valid_items = [it for it in rss_items if "error" not in it]
# 🆕 预过滤

# Step 2: search_web 后（同理）

    # 🆕 预过滤
    new_search_items, _ = _prefilter_against_history(search_items)
    if new_search_items:
        search_supplemented = True
        collected_items.extend(new_search_items)
```

```python
# models/vector_store.py 新增方法
def search_by_embedding(
    self,
    query_embedding: list[float],
    n_results: int = 5,
    collection: str = COLLECTION_FEED_ITEMS,
) -> dict:
    """按原始 embedding 向量查询（不经过 ChromaDB 内置 embedding_fn）。

    用于预过滤场景：先用 bge-small 生成向量，再用此方法查 ChromaDB。
    """
    col = self.get_collection(collection)
    return col.query(query_embeddings=[query_embedding], n_results=n_results)
```

### 改动 4 详细：VectorStore 新增 `upsert_items` 方法

```python
# models/vector_store.py 新增方法
def upsert_items(
    self,
    collection: str,
    ids: list[str],
    documents: list[str],
    metadatas: list[dict],
    embeddings: Optional[list[list[float]]] = None,
):
    """幂等写入条目向量（upsert 避免重复 ID 报错）。

    用于 update_memory_node：已存在的条目 ID 静默更新，新条目新增。
    """
    col = self.get_collection(collection)
    if embeddings is None and self.embedding_fn is not None:
        embeddings = self.embedding_fn(documents)
        if hasattr(embeddings, 'tolist'):
            embeddings = embeddings.tolist()
    col.upsert(ids=ids, documents=documents, metadatas=metadatas, embeddings=embeddings)
```

| ChromaDB feed_items 体积 | 0 条 | 每天 ~30-50 条 | 30 天 ~1500 条，可忽略 |

### 为什么省的是 rank_items 而不是 dedup？

当前 `deduplicate()` 已经有一套高效的三区间策略（高阈值直接判 + LLM 裁决上限），即使 62 条全部进入，LLM 裁决最多也只有 20 次。真正的瓶颈在 `rank_items`：每条都要 LLM 打分排序。减少条目数 = 减少 rank_items 的 LLM token 消耗和时间。

### 与现有 `ranking.dedup_threshold` 的关系

| 参数 | 用途 | 阈值 | 层级 |
|------|------|------|------|
| `prefilter.similarity_threshold` | 跨批次预过滤（新） | 0.92 | Collection→Ranking 之间 |
| `ranking.dedup_threshold` | 批次内直接判重 | 0.88 | Ranking Agent 内 dedup 工具 |
| `ranking.dedup_llm_lower` | 批次内 LLM 裁决下限 | 0.70 | Ranking Agent 内 dedup 工具 |

预过滤用更高阈值（0.92 > 0.88），确保只拦截"几乎确定是同一篇"的条目。中间区间（0.70-0.92）的模糊条目仍然进入 Ranking Agent，由 LLM 做最终裁决。

## 七、风险与注意事项

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| **阈值过高漏掉重复** | 中 | 低 — 漏掉的重复进入 Ranking 仍会被 dedup 拦截 | 监控 `prefilter_discarded_count` 指标，若长期 < 10% 则适当降低阈值 |
| **阈值过低误杀新内容** | 低 | 中 — 用户漏掉有价值的新闻 | 0.92 是保守阈值，且预过滤仅看标题+摘要，不涉及全文理解 |
| **ChromaDB 查询延迟** | 低 | 低 — 62 次查询约 1-2s | ChromaDB ANN 索引查询 < 10ms/次；若后续条目量极大可改为批量查询 |
| **feed_items 向量膨胀** | 低 | 低 — 30 天约 1500 条 | 定时清理（30 天过期）+ 监控 `feed_items` count |
| **bge-small 精度不足** | 极低 | 低 | 现有 dedup 用的就是同一个模型，标题+摘要级别重复判断精度足够 |
| **embedding 生成增加 CPU 开销** | 低 | 极低 | 6.73ms/条 × 62 = 0.4s，相比节省的 LLM 时间可忽略 |

## 九、实际实施差异（vs 原始规划）

> 以下两项是在实施过程中发现的规划盲区，实际代码与原始规划有偏离。

### 9.1 差异 1：编码文本从 `title+summary` 改为纯 `title`

**原始规划**：用 `f"{title} {summary}"` 编码写入 ChromaDB 和查询。

**实际问题**：`summary` 字段在向量去重中引入噪音，稀释标题核心语义。且预过滤是 `feed_items` 集合**唯一实际读取场景**，标题语义已足够判断重复。

**实际实施**：
- 写入端（`main_agent.py`）：`text = title.strip()`
- 查询端（`collection_agent.py`）：`text = title.strip()`
- 两边统一保证向量空间一致

### 9.2 差异 2：写入范围从 `ranked_items` 改为全量 `collected_items`

**原始规划**：仅写入 `ranked_items`（top 10）到 ChromaDB。

**实际问题**：这导致 84% 的条目（53/63）从未写入历史向量库，下次采集时无法被预过滤拦截。用户反馈"每次采集都是固定 63 条"——这就是根因。

**实际实施**：
```python
collected_items = state.get("collected_items", [])
items_to_write = collected_items if collected_items else ranked_items
```
- 优先写入全量 `collected_items`
- `collected_items` 为空时回退到 `ranked_items`（兼容性保护）

### 9.3 差异 3：幂等 ID 从 UUID 改为 SHA256 URL+title hash

**原始规划**：用 `uuid.uuid4()` 作为 ChromaDB ID。

**实际问题**：每次写入都生成新 UUID，同一内容多次执行会产生重复向量。

**实际实施**：
```python
content_key = f"{url}|{title}"
item_hash = hashlib.sha256(content_key.encode("utf-8")).hexdigest()[:32]
```
- 同 URL+title 产生相同 ID，配合 `upsert_items` 实现真正幂等

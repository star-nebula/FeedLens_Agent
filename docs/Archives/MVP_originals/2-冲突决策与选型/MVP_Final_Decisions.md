# FeedLens MVP 最终决策清单

> **本文件为 FeedLens 项目 MVP 阶段的权威决策参考，由 CTO 与产品 VP 联合拍板。**
> 整合来源：TRAE 决策清单（24项）+ WorkBuddy 决策清单（18项），经交叉比对与项目上下文校准后形成。

---

## 整合说明

### 两份清单的关系

| 维度 | TRAE（24项） | WorkBuddy（18项） |
|------|-------------|-----------------|
| 覆盖面 | 更广——含认证/推送渠道/ORM/偏好算法/RSS工具/搜索服务/去重阶段数等实现细节 | 更精——聚焦核心产品逻辑与技术架构分歧 |
| 决策深度 | 1句理由 | 3句深度分析 + 项目上下文引用 |
| 阵营标注 | 仅列文档数量 | 列具体文档名 + 章节引用 |

### 分歧裁决原则

两份清单在 **7 处核心决策上存在分歧**，裁决依据如下：

1. **项目描述优先**：用户提供的项目定位与场景描述是第一优先级依据。例如项目描述"同一事件不同来源的报道只保留一条"直接解决了去重结果处理的分歧。
2. **逻辑自洽优先**：冷启动阶段没有偏好数据，偏好优先排序在逻辑上不成立，因此排序权重必须采用"冷启动相似度优先→有反馈后偏好优先"的动态方案。
3. **MVP 成本最低优先**：在逻辑都成立的情况下，选开发/运营成本最低的方案。

### 整合策略

- **合并重叠项**：TRAE 的"推送触发模式"与"推送渠道"合并为 WB 的"推送机制"；TRAE 的"去重技术路线"+"向量去重+LLM精判"+"去重阶段数"+"去重阈值"合并为 WB 的综合去重策略。
- **保留独有项**：TRAE 独有的认证/推送渠道/ORM/偏好算法/RSS工具/搜索服务等实现细节，纳入"实现选型"板块（无真实分歧，直接拍板）。
- **WB 独有项纳入**：WB 的"简报输出风格"是 TRAE 未覆盖的重要分歧，纳入冲突决策。

## 一、冲突决策清单（有真实分歧，需深度分析）

---

### P1：简报生成驱动模式（类型：产品逻辑）

- **阵营A（流程驱动）**：按预设工作流每日定时采集并生成日报。依据：Codex_DeepSeek 2.1节 / DeepSeek 2.2节 / GLM / Kimi / Perplexity / Qwen / TRAE 共 8 份文档均有明确的阶段划分和调度触发。
- **阵营B（Goal 驱动）**：Agent 围绕用户长期目标自主决定何时搜索、何时推送。依据：GPT 第三节 Planner 输出 JSON action + Reflection→Planner 自主循环。
- **🏆 最终决策：阵营A（流程驱动）**
- **💡 决策理由**：项目定位强调"定时执行"而非"自主规划"，MVP 核心验证点是"个性化信息简报是否有用户价值"而非"自主 Agent 规划能力"。流程驱动开发成本远低于 Goal 驱动（省去 Planner 决策循环 + Reflection 重试逻辑），2 周内可跑通闭环。Goal 驱动作为 v2.0 差异化升级路径保留。

---

### P2：推送机制与渠道（类型：产品逻辑）

- **阵营A（定时推送）**：每日固定时间推送简报，或通过 Streamlit 手动查看。依据：8 份文档均在阶段末尾触发推送节点。
- **阵营B（自主推送）**：Planner 判断"重大事件立即推送，普通事件积累成日报"。依据：GPT 第四节 PushNow action。
- **🏆 最终决策：定时推送 + 重大事件破例立即推送（应用内渠道）**
- **💡 决策理由**：MVP 先实现固定时间每日推送（仅需 APScheduler cron job），开发成本最低。项目定位中的"自主规划"能力通过破例推送体现：当单条 score > 0.85 且 freshness < 2h 时立即推送。推送渠道为 Streamlit 应用内通知（8/9 文档共识），无需对接第三方平台（Telegram/微信等是 v2.0 范围）。

---

### P3：用户配置粒度（类型：产品逻辑）

- **阵营A（多维度配置）**：用户配置关注领域、关键词、RSS 源列表。依据：Codex_Mimo 3.1节 / DeepSeek 4.1节 / Kimi 2.2节。
- **阵营B（仅需 Goal）**：用户只需输入一个长期目标文本。依据：GPT 文档"用户只需告诉我她的 Goal"。
- **🏆 最终决策：阵营B（Goal 文本输入）+ 后端 LLM 提取结构化字段**
- **💡 决策理由**：项目场景描述"用户设定关注的领域"可通过 Goal 文本实现——输入"我想跟踪 AI Agent 和新能源车的最新进展"，LLM 一次调用提取 topics/keywords/sources，存入 user_profile 表。用户体验远优于结构化表单（减少流失），开发成本仅 +1 个 LLM 调用，完全兼容阵营A的后端数据结构。

---

### P4：去重结果处理（类型：产品逻辑）

- **阵营A（折叠分组）**：同事件不同角度归入同一 cluster，简报中展示为"相关报道"折叠组。依据：GLM 6.2节 / Codex_Mimo 6.1节。
- **阵营B（计数标注）**：保留一篇代表，标注"还有 N 篇类似报道"。依据：DeepSeek 6.2节。
- **🏆 最终决策：阵营B（计数标注）**
- **💡 决策理由**：项目描述明确指出"同一事件不同来源的报道只保留一条"，直接锁定阵营B。Streamlit 实现计数标注仅需文本模板 `{title}（还有 {n} 篇类似报道）`，零额外前端成本；折叠分组需自定义 CSS/JS 组件（+0.5天），MVP 不值得。

---

### P5：搜索采集角色（类型：产品逻辑）

- **阵营A（搜索为并列通道）**：搜索与 RSS 并行采集。依据：Codex_Mimo 2.1节"三路并行" / Kimi 2.1节点图 / TRAE 3.2节。
- **阵营B（搜索为补充）**：搜索仅在 RSS 条目不足时触发。依据：DeepSeek 2.2节 collect_sources fallback / 其余文档未明确并行。
- **🏆 最终决策：阵营B（RSS 为主 + 搜索补充）**
- **💡 决策理由**：项目场景描述"从多个信息源（RSS、搜索引擎）采集"中 RSS 排在前，搜索是补充角色。"并列通道"每次运行都调用搜索 API（+成本 +速率限制风险），"补充"仅在去重后条目 < 5 时触发搜索，运营成本显著更低。开发成本相当，但 RSS 并行多源采集（免费）已能覆盖大部分场景。

---

### P6：简报输出风格（类型：产品逻辑 → 偏技术）

- **阵营A（结构化 JSON）**：LLM 输出 JSON 再渲染为 Markdown。依据：GLM 5.2节 / Kimi 5.3节 brief_quality 评分。
- **阵营B（直接 Markdown）**：LLM 直接输出 Markdown。依据：Codex_DeepSeek / DeepSeek / Perplexity / Qwen / TRAE。
- **🏆 最终决策：阵营A（结构化 JSON 输出）**
- **💡 决策理由**：项目描述要求"结构化简报（分类组织 + 重要性标注 + 来源引用）"，这天然对应 JSON schema。JSON 输出可 JSON Schema 校验、可自动质量评分触发重试（Kimi brief_quality < 0.7）、前端渲染与生成解耦。额外成本仅 +1 个 system prompt 段落约束输出格式，换来可维护性显著提升。

---

### P7：用户反馈选项（类型：产品逻辑）

- **阵营A（多元反馈）**：like / dislike / irrelevant / valuable 四级。依据：Kimi 4.3节 / Codex_Mimo / Qwen。
- **阵营B（二元反馈）**：仅 like / dislike。依据：Codex_DeepSeek / DeepSeek / Perplexity。
- **🏆 最终决策：三级反馈（like / dislike / irrelevant）**
- **💡 决策理由**：项目场景描述明确区分了"这条不相关"和"这条有价值"两种不同信号——"irrelevant"表示"不属于我的关注领域"，与"dislike"（属于但质量差）语义不同。有了 irrelevant 信号，系统可从候选集移除整类内容，偏好收敛速度远快于二元反馈。valuable 与 like 语义重叠，舍去。前端仅需 3 个按钮（+0.5h 开发），收益显著。

---

### P8：用户认证模式（类型：产品逻辑）

- **阵营A（需要认证）**：依据：TRAE 文档用户认证章节。
- **阵营B（不需要）**：依据：其余 8 份文档均为单用户 MVP。
- **🏆 最终决策：阵营B（单用户，不需要认证）**
- **💡 决策理由**：MVP 核心验证点是"个性化筛选是否有价值"，单用户模式足以验证。认证增加开发成本（+登录页 + session 管理 + 权限校验），且与核心假设无关。多用户认证作为 v2.0 范围，届时 user_id 外键已预留。

---

### T1：MCP 传输模式（类型：技术架构）

- **Push 部署**：stdio（5份：DeepSeek/GLM/Kimi/TRAE/Codex_Mimo）vs SSE（3份：Codex_DeepSeek/Perplexity/Qwen）
- **Search 部署**：SSE（7份共识）vs stdio（0份采用）
- **🏆 最终决策：Push 用 stdio，Search 用 SSE**
- **💡 决策理由**：Push 用 stdio——推送是本地操作（Streamlit 通知），子进程随主进程启停，无需端口管理；SSE 优势是跨机器调用，但 MVP 所有组件同机运行。Search 用 SSE——搜索对接外部 API，SSE 支持流式响应且 7/7 文档共识，无争议。

---

### T2：向量去重策略（阈值 + 技术路线 + 阶段数）（类型：技术架构）

- **阈值分歧**：高阈值 ≥0.90（DeepSeek/GLM/Kimi/TRAE）vs 中阈值 0.85-0.88（Codex_DeepSeek/Codex_Mimo/Perplexity/Qwen）
- **技术路线分歧**：纯向量阈值判定（Codex_DeepSeek/DeepSeek/GPT）vs 多信号组合（Codex_Mimo/GLM/Qwen/Perplexity）
- **LLM 精判分歧**：阈值内一律去重（多数文档）vs 模糊区间 LLM 裁决（Qwen/Kimi/Perplexity）
- **🏆 最终决策：0.88 初始阈值 + 纯向量 + 模糊区间（0.70-0.88）LLM 裁决 + 校准脚本**
- **💡 决策理由**：0.88 是双方重叠最多的数值（4份文档用 0.88），争议最小。纯向量去重开发成本最低，但"智能去重"需要覆盖"苹果发布新 iPhone" vs "Apple 推出 iPhone 16"这类跨语言边界场景——仅在 0.70-0.88 模糊区间调用 LLM 做二元判断（高置信 ≥0.88 直接去重，低置信 <0.70 直接放过），调用量可控。校准脚本（GLM calibrate_dedup.py）让阈值成为可优化的系统参数而非硬编码。

---

### T3：排序权重配置（类型：技术架构）

- **阵营A（偏好优先 w_pref ≥ 0.30）**：Codex_DeepSeek / Codex_Mimo / GLM / Kimi。
- **阵营B（相似度优先 w_sim ≥ 0.35）**：GPT / DeepSeek / Perplexity / TRAE / Qwen。
- **🏆 最终决策：冷启动相似度优先 → 有反馈后动态切换偏好优先**
- **💡 决策理由**：冷启动阶段（用户无反馈历史）偏好权重无数据支撑，逻辑上只能用相似度排序。当反馈 ≥ 3 条时自动切换偏好优先公式（借鉴 Codex_DeepSeek w3 从 0.10→0.40 的动态调整）。初始公式采用 GPT 四因子（relevance + preference + importance + freshness），有反馈后偏好权重提升至 0.30+。

---

### T4：Embedding 模型选择（类型：技术架构）

- **阵营A（本地 sentence-transformers）**：Codex_DeepSeek / GLM / TRAE / Codex_Mimo / Kimi（5份）。
- **阵营B（API 调用）**：DeepSeek / Qwen（2份）。
- **🏆 最终决策：阵营A（本地 bge-small-zh-v1.5）**
- **💡 决策理由**：本地模型一次下载、无限调用、无速率限制，MVP 开发调试体验远优于 API（频繁重跑不计费）。bge-small-zh-v1.5 中文语义相似度表现优秀，153M 参数，推理 ~50ms/条。预留 API 切换接口（Qwen 双轨思路），用量稳定后再评估。

---

### T5：MCP Server 数量与工具划分（类型：技术架构）

- **阵营A（3+ 个 MCP）**：DeepSeek(3) / Perplexity(5) / TRAE(3)。
- **阵营B（2 个 MCP）**：Codex_DeepSeek / Codex_Mimo / GLM / Kimi / Qwen（5份）。
- **🏆 最终决策：阵营B（2 个 MCP：search SSE + push stdio）+ DB/向量操作改为 FC**
- **💡 决策理由**：MVP 所有组件用 Python 编写，DB/向量操作是标准 SDK 调用，封装为 MCP 只增加进程间通信开销和代码量（每个 MCP Server ≈ +200 行实现代码 + 进程管理逻辑）。仅保留 2 个 MCP：search_web（外部 API，适合独立服务）和 push_notification（涉及通知权限，适合独立进程）。

---

### T6：短期记忆窗口（类型：技术架构）

- **阵营A（10 轮）**：Codex_DeepSeek。
- **阵营B（15 轮）**：Codex_Mimo / DeepSeek / Kimi。
- **阵营C（10-20 滑动）**：Kimi 描述。
- **🏆 最终决策：15 轮 + 超窗压缩写入情节记忆**
- **💡 决策理由**：一个完整工作流（采集→去重→排序→生成→推送）约占 5-8 轮，10 轮过紧可能截断上下文。15 轮有余量且是多数文档默认值。采纳 Kimi 的超窗压缩设计：超过 15 轴时将前 10 轮压缩为情节记忆写入 SQLite，短期记忆始终 15 轴窗口但长期上下文不丢失（+1 个 LLM 压缩调用，成本可控）。

---

### T7：时间衰减函数（类型：技术架构）

- **阵营A（指数衰减 exp(-λ·Δt)）**：GPT / Codex_Mimo / DeepSeek / Kimi / Qwen。
- **阵营B（半衰期公式 exp(-Δt/τ)）**：GLM(τ=24h) / TRAE(HALF_LIFE=24h)。
- **🏆 最终决策：阵营B（半衰期公式），τ=24h**
- **💡 决策理由**：数学完全等价（λ = 1/τ），但半衰期形式对 PM 和非技术人员更直观："24 小时前的内容得分减半"比"λ=0.05 的指数衰减"容易理解和调参。τ 设为可配置参数（默认 24h），未来调整衰减速度改配置即可，无需改代码。

---

### T8：反思重试次数（类型：技术架构）

- **阵营A（2-3 次）**：GLM(max 2) / Codex_Mimo(max 2) / Kimi(max_retry=3)。
- **阵营B（1 次或不明确）**：GPT / Codex_DeepSeek。
- **🏆 最终决策：最多 2 次重试，brief_quality < 0.7 触发**
- **💡 决策理由**：0 次重试无质量保障，1 次可能不够（首次反思后仍可能不合格），3 次边际收益递减且成本高（每次 = 1 次 LLM 调用）。2 次是实验上合理的默认值。采用 Kimi 的 brief_quality 评分触发机制：生成简报后自动评分，< 0.7 触发重试，最多 2 次；2 次后仍 < 0.7 则接受当前最佳结果并记录日志。

---

### T9：向量库交互方式（类型：技术架构）

- **阵营A（FC 直接调用 ChromaDB SDK）**：6 份文档。
- **阵营B（封装为 MCP Server）**：TRAE / DeepSeek(db) / Qwen。
- **🏆 最终决策：阵营A（FC 直接调用）**
- **💡 决策理由**：向量库操作（add/get/search）是 ChromaDB 标准 SDK 调用，无需独立进程、无需跨语言复用。FC 直接调用开发成本更低（import chromadb vs 写 MCP Server 实现 + 进程管理），延迟更低（无 IPC 开销）。未来需要独立部署向量库时再封装为 MCP SSE，改动纯增量。

---

### T10：部署方案（类型：技术架构）

- **阵营A（本地开发运行）**：7 份文档。
- **阵营B（Docker Compose）**：GLM / Perplexity / DeepSeek。
- **🏆 最终决策：本地开发运行 for MVP + Docker Compose for v1.0 交付**
- **💡 决策理由**：MVP 开发阶段需要频繁改代码、重启服务、看日志，Docker 每次改代码要 rebuild/mount+重启，迭代速度显著低于本地运行。v1.0 交付给用户时必须提供 Docker Compose 一键启动（GLM 的设计保留），但 MVP 开发阶段不用。

| I10 | **Agent 编排框架** | LangGraph StateGraph | 9/9 文档共识；无分歧 |

## 三、决策汇总表

| # | 冲突/选型 | 最终决策 | 核心理由 | 来源 |

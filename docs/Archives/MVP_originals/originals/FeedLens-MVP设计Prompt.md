我要做一个 **Agent 项目**作为我的简历项目（目标岗位：AI 应用开发）。请你帮我设计 MVP 方案。

## 项目名称：FeedLens — 智能信息简报 Agent

### 项目定位

一个**主动式**信息聚合 Agent。它不是被动问答，而是能**定时自主采集、智能去重、偏好排序、生成简报并推送**给用户。核心差异化：展示 Agent 的「自主规划 + 定时执行 + 个性化筛选」能力。

### 核心用户场景

- 用户设定关注的领域（如「AI Agent」「新能源车」「跨境电商」）
- Agent 每天定时从多个信息源（RSS、搜索引擎）采集最新内容
- 自动去重（同一事件不同来源的报道只保留一条）
- 根据用户偏好排序（用户历史点赞/踩的内容会影响后续排序权重）
- 生成结构化简报（分类组织 + 重要性标注 + 来源引用），推送给用户
- 用户反馈「这条不相关」「这条有价值」→ Agent 学习并调整后续筛选

### 我对 Agent 架构的理解（作为设计约束）

请你基于以下 Agent 架构知识来设计方案，不要脱离这个框架另搞一套：

#### 1. Agent 的核心组成

- **感知层（Perception）**：文本输入、结构化数据、工具返回结果等作为 Agent 的感知输入
- **大脑（Brain）**：LLM 负责——意图理解、推理决策、工具调用判断
- **工具（Tools）**：信息获取（搜索/RSS/文档）、计算执行（代码解释器）、内容生成（摘要/简报）、系统交互（推送/通知）
- **记忆（Memory）**：
  - **短期记忆**：当前对话上下文，受限于模型上下文长度，用滑动窗口管理
  - **长期记忆**：用户偏好、知识积累，通过向量数据库持久化，支持 RAG 检索
  - **情节记忆**：历史任务执行记录，「上次遇到这种情况怎么处理的」，帮助从经验中学习
  - **语义记忆**：抽象知识和事实，可通过 RAG 动态补充
- **规划（Planning）**：
  - **ReAct 模式**：思考→行动→观察→再思考的循环，适合需要工具调用的动态任务
  - **反思（Reflection）**：任务完成后对输出进行批判性审查并修正

#### 2. 工程实现框架

- **LangGraph**：用 StateGraph 定义节点和边，状态是共享数据结构（TypedDict），节点间通过状态传递数据
- **工具调用方式**：Agent 有两种工具调用途径——
  - **Function Calling（原生工具调用）**：LLM 自主判断是否需要调用工具，输出标准调用格式 `{name, parameters}`，适用于逻辑简单、参数明确的工具（如：RSS 采集、文本摘要、简单查询）
  - **MCP（Model Context Protocol）**：通过标准化的 MCP 协议连接外部工具服务器，适用于需要独立部署、跨进程复用、或已有 MCP Server 的工具（如：搜索服务、推送通知、数据库操作）。MCP 的优势是工具与 Agent 解耦、可渐进式加载、支持跨 Agent 共享
  - **设计要求**：请根据每个工具的复杂度和复用场景，**合理选择使用 Function Calling 还是 MCP**，并在工具清单中标注选择理由。不要一刀切只用一种
- **对话历史管理**：滑动窗口（10-20轮）、基于时间衰减、基于相关性选择、增量总结压缩
- **Harness 工程**：session（会话）→ turn（一次流程）→ event（事件），工具注册包含 skills（渐进式加载）、MCP、子 Agent

### 设计要求

请你输出的 MVP 方案需包含以下内容：

1. **系统架构图**：画出 FeedLens 的整体架构，标注每个模块属于 Agent 的哪一层（感知/大脑/工具/记忆/规划）
2. **Agent 工作流设计**：用 LangGraph StateGraph 描述完整的节点和边，定义 State 的 TypedDict 字段
3. **工具清单**：列出 MVP 需要实现的每个 Tool，对于每个工具——
   - 给出 name、description、parameters 定义
   - **标注调用方式**：Function Calling 还是 MCP，并给出选择理由
   - 如果是 MCP，说明该工具作为 MCP Server 的部署方式（stdio / SSE）和接口定义
4. **记忆系统设计**：短期/长期/情节记忆分别用什么存储、存什么内容、怎么检索、怎么更新
5. **排序算法设计**：信息条目排序的打分公式和权重说明（向量相似度 + 时间衰减 + 偏好权重）
6. **去重策略设计**：向量去重的阈值设定、校准方法、如何区分「同一事件不同角度」和「真正重复」
7. **数据模型**：SQLite 表结构设计（用户、信息源、条目、反馈、简报）
8. **技术栈选择**：给出每个层级的具体技术选择和理由
9. **MVP 范围界定**：列出 MVP 必须实现的功能 vs 后续迭代才加的功能
10. **阶段性目标与任务拆解**：不要按「每天」拆，而是按「阶段」拆，每个阶段是一个独立的、可验证的交付单元。每个阶段需标注：
    - 阶段目标（这个阶段要达成什么）
    - 交付物（具体产出什么，怎么验证它是对的）
    - 关键任务列表
    - 依赖的前一阶段
    - 预估复杂度（低/中/高）

### 约束条件

- 目标岗位是 AI 应用开发，方案要体现工程落地能力，不是纯学术
- 我会用 AI 辅助完成项目实现，所以不需要估算工期，只需要清晰的阶段性目标和交付物定义

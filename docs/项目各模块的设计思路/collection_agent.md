# Collection Agent（采集）设计思路

> 内容采集 Agent，负责从 RSS 源 + 搜索引擎获取内容，支持 ReAct 和 Pipeline 双模式。

---

## 整体定位

Collection Agent 是管线的**第一环**，负责：
- 从 RSS 源并行采集内容
- 搜索补充（RSS 不足时）
- 向量预过滤（跨批次去重）
- 字段标准化

---

## 完整流程

### Pipeline 模式（默认，无 LLM 参与）

```
输入: goal_text + structured_goal
  │
  ▼
┌─ ① 获取 RSS 源列表 ─────────────────────────────────────┐
│  · 数据库 sources 表（is_active=1，按 authority_score 排序）│
│  · 用户 structured_goal.preferred_sources               │
│  · 默认源列表（7 个硬编码源）                              │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─ ② fetch_rss — 并行采集 ────────────────────────────────┐
│  · 多线程并发请求 RSS 源（max_workers=5）                  │
│  · 过滤错误条目                                           │
│  · 🆕 向量预过滤：查询 ChromaDB 历史条目，丢弃余弦相似度    │
│     ≥ 0.92 的条目（跨批次去重）                            │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─ ③ search_web — 搜索补充（仅当 RSS < 阈值时触发）─────────┐
│  · 从 goal 构建搜索查询词（topics + keywords）             │
│  · 通过 MCP search 客户端调用搜索引擎                      │
│  · 同样经过向量预过滤                                      │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─ ④ normalize_items — 字段标准化 ─────────────────────────┐
│  · 统一标题/时间/来源/ID 字段格式                          │
│  · 补充 fetched_at 时间戳                                 │
└─────────────────────────────────────────────────────────┘
  │
  ▼
输出: collected_items + search_supplemented + collection_summary
```

### ReAct 模式（兼容模式，LLM 自主决策）

```
LLM 思考 → 调用 fetch_rss → 观察结果
  → 采集量 < 5？→ 调用 search_web
  → 调用 normalize_items → 调用 finish_task
```

---

## 向量预过滤（_prefilter_against_history）

```
对每个条目：
  1. bge-small 编码 title + summary → 向量
  2. 在 ChromaDB feed_items 集合中查最相似历史条目
  3. 余弦相似度 ≥ 0.92 → 丢弃（历史已有）
  4. 余弦相似度 < 0.92 → 保留（新数据）

降级策略：任何异常都透传全部条目，不阻塞管线
冷启动：ChromaDB 为空时透传全部
```

## RSS 源获取优先级

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | `sources` 表（is_active=1） | 数据库管理，按权威分排序 |
| 2 | `structured_goal.preferred_sources` | 用户偏好配置 |
| 3 | 硬编码默认源 | 7 个源兜底（36kr/知乎/InfoQ/Solidot 等） |

---

## 关键设计决策

| 决策 | 做法 | 理由 |
|------|------|------|
| **Pipeline 默认模式** | 固定流水线，不经过 LLM 思考 | 采集是确定性流程，省 API 成本 |
| **向量预过滤** | 采集阶段即过滤历史重复 | 减少后续去重和排序的负担 |
| **降级透传** | 预过滤异常时全部保留 | 宁可多采不可漏采 |
| **双模式切换** | 配置项 `collection_mode` 控制 | 保留 ReAct 兼容性，默认 Pipeline |

---

## 一句话总结

> 获取 RSS 源列表 → 并行采集 → 向量预过滤跨批次去重 → RSS 不足时搜索补充 → 字段标准化 → 输出。

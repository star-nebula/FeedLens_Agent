# Ranking Agent（排序）设计思路

> 内容排序 Agent，负责向量去重 + 多因子偏好排序，是决定"用户看到什么"的核心环节。

---

## 整体定位

Ranking Agent 是管线的**第二环**，接收采集结果，输出排序后的条目列表。核心任务：
- 向量相似度去重（两阶段：硬阈值 + LLM 裁决）
- 多因子加权排序（相似度 / 时效性 / 用户偏好 / 重要性）
- 冷启动 vs 偏好模式自适应切换

---

## 完整流程

```
输入: collected_items + user_id + goal_embedding
  │
  ▼
┌─ ① 加载用户数据 ─────────────────────────────────────────┐
│  · 从 ChromaDB 加载用户偏好向量（v_like / v_dislike）       │
│  · 从 SQLite 加载近 30 天反馈历史                          │
│  · 从 config 加载权重配置                                  │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─ ② deduplicate — 向量去重 ───────────────────────────────┐
│                                                         │
│  两阶段策略：                                             │
│                                                         │
│  ┌─ 硬阈值判定 ─────────────────────────────────┐        │
│  │  余弦相似度 ≥ 0.88 → 直接判重，丢弃            │        │
│  │  余弦相似度 ≤ 0.70 → 直接保留                 │        │
│  └──────────────────────────────────────────────┘        │
│                                                         │
│  ┌─ LLM 裁决（中间区间 0.70~0.88）─────────────┐         │
│  │  LLM 判断两条是否语义重复                     │         │
│  │  最多裁决 20 对，超出则按阈值截断              │         │
│  └──────────────────────────────────────────────┘        │
│                                                         │
│  输出: unique_items + duplicate_pairs                    │
│  同时写入 SQLite item_relations 表                       │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─ ③ rank_items — 多因子排序 ──────────────────────────────┐
│                                                         │
│  ┌─ 预筛（时间窗口）────────────────────────────┐        │
│  │  默认 72h（3 天）内的条目                      │        │
│  │  expand_threshold 模式 → 336h（14 天）         │        │
│  └──────────────────────────────────────────────┘        │
│                                                         │
│  ┌─ 模式判断 ──────────────────────────────────┐        │
│  │  反馈数 < 3 → 冷启动模式                      │        │
│  │  反馈数 ≥ 3 → 偏好模式                        │        │
│  └──────────────────────────────────────────────┘        │
│                                                         │
│  ┌─ 四维评分 ──────────────────────────────────┐        │
│  │                                             │        │
│  │  similarity（与 goal 的余弦相似度）           │        │
│  │    冷启动: 40%  │  偏好模式: 30%              │        │
│  │                                             │        │
│  │  recency（时效性，指数衰减）                  │        │
│  │    冷启动: 25%  │  偏好模式: 20%              │        │
│  │    score = e^(-hours/24)                    │        │
│  │                                             │        │
│  │  preference（用户偏好匹配）                   │        │
│  │    冷启动: 10%  │  偏好模式: 40%              │        │
│  │    冷启动时代理为 similarity                  │        │
│  │    偏好模式: like_sim - dislike_sim          │        │
│  │    叠加反馈偏置（like:+0.15, dislike:-0.10）  │        │
│  │                                             │        │
│  │  importance（重要性评分，LLM 评估）            │        │
│  │    冷启动: 25%  │  偏好模式: 10%              │        │
│  │                                             │        │
│  │  final_score = Σ(weight × score)            │        │
│  └──────────────────────────────────────────────┘        │
│                                                         │
│  按 final_score 降序排列，取 top N（默认 10，expand 20）    │
└─────────────────────────────────────────────────────────┘
  │
  ▼
输出: ranked_items + ranking_detail (含各因子得分明细)
```

---

## 冷启动 vs 偏好模式权重对比

| 因子 | 冷启动（反馈 < 3） | 偏好模式（反馈 ≥ 3） | 说明 |
|------|-------------------|---------------------|------|
| similarity | 40% | 30% | 冷启动更依赖目标匹配 |
| recency | 25% | 20% | 冷启动更看重时效 |
| preference | 10% | **40%** | 偏好模式核心因子 |
| importance | 25% | 10% | 冷启动更依赖客观重要性 |

---

## 关键设计决策

| 决策 | 做法 | 理由 |
|------|------|------|
| **两阶段去重** | 硬阈值 + LLM 裁决中间区间 | 简单 case 快速判定，复杂 case 才用 LLM |
| **冷启动/偏好双模式** | 反馈数阈值切换权重 | 新用户无偏好数据时降级为相似度代理 |
| **偏好向量正负分离** | like 向量 - dislike 向量 | 比单一偏好更精准，知道用户不喜欢什么 |
| **时间窗口预筛** | 默认 72h，expand 模式 336h | 控制排序范围，避免陈旧内容干扰 |
| **预筛过严检测** | 采集充足但排序极少 → 标记 | 避免误判为排序失败触发完整重跑 |

---

## 一句话总结

> 加载用户偏好 → 两阶段向量去重（硬阈值 + LLM 裁决） → 冷/热模式切换 → 四因子加权打分 → 按分数降序输出 top N。

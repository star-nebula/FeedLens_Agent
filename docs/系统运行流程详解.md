# FeedLens 系统运行流程详解

> 文档版本：v1.0 | 更新日期：2026-06-24

---

## 一、系统概述

**FeedLens** 是一个基于 LangGraph + LLM 的智能信息简报系统。核心能力是：从 RSS 源采集新闻/资讯条目，经过去重、多因子排序、简报生成和质量审查，最终推送给用户。系统采用多 Agent 协作架构，通过 Planner（编排器）+ Router（路由器）实现 LLM 自主决策的 ReAct 循环。

**技术栈**：Streamlit（UI） + LangGraph（Agent 编排） + DeepSeek（LLM） + ChromaDB（向量存储） + SQLite（关系数据） + bge-small-zh-v1.5（Embedding）

## 三、启动流程

### 3.1 入口：`streamlit run app.py`

`app.py` 是 Streamlit 前端入口，提供 6 个页面：

| 页面 | 路由 Key | 功能 |
|------|----------|------|
| 首页 | `home_page` | 查看简报列表、手动触发管线、提交反馈 |
| Goal 设置 | `goal_page` | 设置用户信息需求（目标文本、关键词等） |
| RSS 源管理 | `sources_page` | 管理 RSS 数据源 |
| 反馈记录 | `feedback_page` | 查看历史反馈 |
| 执行日志 | `logs_page` | 查看 Agent 执行日志 |
| 执行仪表盘 | `dashboard_page` | 可视化执行统计 |

### 3.2 手动触发管线

当用户点击"立即运行"按钮时，首页调用 `_run_pipeline_with_tee("manual")`，该函数会：

1. 启动一个**子进程**运行 `python -m utils.pipeline_runner --trigger manual`
2. 子进程的输出通过管道同时写入 `data/pipeline.log` 文件和终端 stderr
3. 主进程（Streamlit UI）不阻塞，子进程独立运行

```python
# ui/pages/home_page.py 关键代码
def _run_pipeline_with_tee(trigger: str = "manual"):
    proc = subprocess.Popen(
        [sys.executable, "-m", "utils.pipeline_runner", "--trigger", trigger],
        stdout=subprocess.PIPE,
        stderr=subprocess.STDOUT,
    )
    # 后台线程读取输出
    threading.Thread(target=_reader, daemon=True).start()
```

### 3.3 Pipeline Runner 初始化

`utils/pipeline_runner.py` 的 `run_agent_pipeline()` 负责：

1. **执行栅栏**：获取文件锁（`execution_fence`），防止定时/手动/破例推送并发执行
2. 从 SQLite `users` 表读取用户的 `goal_text`
3. 生成 `session_id`（格式：`manual_YYYYMMDD_HHMMSS_随机8位hex`）
4. 构建 `FeedLensState` 初始状态并调用主 Agent
5. 计时并返回执行摘要

## 四、主 Agent 工作流（核心管线）

主 Agent 基于 **LangGraph StateGraph** 构建，节点间通过 **router_node** 进行 LLM 驱动的动态路由。

### 4.1 节点清单

| 节点名称 | 函数 | 职责 |
|----------|------|------|
| `understand_intent` | `understand_intent_node` | 理解用户意图，结构化提取 Goal，生成 embedding |
| `planner` | `planner_node` | LLM 编排决策，输出 `sub_agent_plan` |
| `router_node` | `router_node` | 动态路由决策（规则优先 + LLM 兜底） |
| `invoke_sub_agent` | `invoke_sub_agent_node` | 顺序执行子 Agent（Collection → Ranking → Briefing） |
| `observe_results` | `observe_results_node` | 观察评估子 Agent 执行结果 |
| `coordinator_reflect` | `coordinator_reflect_node` | 综合质量审查 |
| `push_notification` | `push_notification_node` | 推送简报 |
| `update_memory` | `update_memory_node` | 写入执行日志、更新偏好向量、保存简报 |

### 4.2 状态定义（FeedLensState）

`agents/state.py` 定义了全局共享状态，包含以下关键字段：

```python
class FeedLensState(TypedDict, total=False):
    # 会话元信息
    session_id: str
    trigger_type: str          # daily_briefing | manual | breaking_news
    user_id: int

    # 用户 Goal
    goal_text: str
    structured_goal: dict      # {topics, keywords, preferred_sources}
    goal_embedding: list[float]

    # 主 Agent 编排控制
    sub_agent_plan: list       # planner 输出的编排计划
    react_cycle_count: int     # ReAct 循环计数（max=3）
    agentic_turn_count: int    # 主循环总轮数
    sub_agent_executed: bool

    # 子 Agent 结果
    collected_items: list      # 采集 Agent 输出
    ranked_items: list         # 排序 Agent 输出
    briefing: dict             # 简报 JSON 内容
    brief_quality: float       # 简报质量评分

    # 观察与审查
    observation_result: dict   # observe_results 输出
    coordinator_observation: dict  # 综合审查结果

    # 路由控制
    router_decision: dict      # {"next_node": "...", "reason": "..."}
    router_history: list       # 路由决策历史
```

### 4.4 关键路由规则

Router 采用**规则优先 + LLM 兜底**策略：

| 优先级 | 条件 | 路由目标 |
|--------|------|----------|
| 1 | `sub_agent_plan` 非空且未执行 | `invoke_sub_agent` |
| 2 | 已执行但未观察 | `observe_results` |
| 3 | 已观察、需重试且未达上限 | `planner`（LLM 重新编排） |
| 4 | 已观察、无需重试 | `coordinator_reflect` |
| 5 | 审查通过 | `push_notification` |
| 6 | 推送完成 | `update_memory` |
| 7 | 超过最大轮数（5轮） | 强制收敛到 `update_memory` |
| 8 | 死循环检测（连续3次同一节点） | 强制收敛 |

## 五、各节点详解

### 5.1 understand_intent（意图理解）

**位置**：`agents/main_agent.py:171`

**职责**：
1. 从 SQLite `users` 表读取用户的结构化偏好（topics, keywords, preferred_sources）
2. 如果用户 `goal_text` 有内容但结构化字段为空，调用 LLM 提取结构化信息
3. 使用 `bge-small-zh-v1.5` 模型生成 `goal_embedding`（用于后续排序的相似度计算）

**输入**：`trigger_type`, `goal_text`, `user_id`
**输出**：`trigger_type`, `structured_goal`, `goal_embedding`, `react_cycle_count=0`, `status="running"`

### 5.2 planner（编排决策）

**位置**：`agents/main_agent.py:266`

**职责**：
1. 检索**记忆系统**：
   - 情节记忆（SQLite `execution_logs`，近7天记录）
   - 长期记忆（ChromaDB `domain_knowledge`，语义检索历史经验）
2. 构建包含当前状态（采集量、排序top_score、简报质量等）的上下文
3. 调用 LLM 决策下一步子 Agent 编排计划
4. LLM 失败时回退到**默认三板斧**（Collection → Ranking → Briefing）

**LLM System Prompt 关键策略**：
- 采集 < 3 条且未补充搜索 → `search_expand`
- 排序 top_score < 0.3 → 考虑 rerank
- 简报条目 < 10 且采集足够 → `expand_threshold`
- 简报质量 < 0.7 → 重试或跳过
- react_cycle >= 2 → 优先收敛

**输出格式**：
```json
{
  "sub_agent_plan": [
    {"agent": "Collection", "params": {"search_expand": {"query": "agents；LLM"}}},
    {"agent": "Ranking", "params": {}},
    {"agent": "Briefing", "params": {}}
  ],
  "reason": "初始状态，采集为空，对目标关键词进行搜索补充采集，然后排序。",
  "push_immediate": false
}
```

### 5.3 router_node（动态路由）

**位置**：`agents/main_agent.py:538`

**职责**：
1. **死循环检测**：检查最近 3 次路由决策是否相同，相同则强制收敛
2. **硬兜底**：超过配置的 `max_turns`（默认5轮）强制结束
3. **规则优先路由**：调用 `_rule_based_router_decision()` 进行确定性路由
4. **LLM 兜底**：规则无法覆盖时路由到 planner 重新编排

**规则路由决策逻辑**：
```python
def _rule_based_router_decision(state):
    # 1. 有未执行计划 → invoke_sub_agent
    # 2. 已执行未观察 → observe_results
    # 3. 需重试且未达上限 → None（需要 LLM 重新编排）
    # 4. 无需重试 → coordinator_reflect
    # 5. 审查通过 → push_notification
    # 6. 审查不通过未达上限 → None
    # 7. 审查不通过已达上限 → push_notification（强制推送）
    # 8. 推送完成 → update_memory
    # 9. 兜底 → update_memory
```

### 5.4 invoke_sub_agent（子 Agent 调用）

**位置**：`agents/main_agent.py:619`

**职责**：
1. 按 `sub_agent_plan` 顺序执行子 Agent
2. 通过 `run_with_isolation` 隔离每个子 Agent，单个失败不阻断后续
3. 将 `params`（如 `search_expand`、`expand_threshold`）注入到子 Agent 的 state 中
4. 子 Agent 成功后合并结果到全局 state
5. 记录各子 Agent 执行状态（success/isolated/not_executed）和耗时

**子 Agent 映射**：
| 名称 | 构建函数 | 结果 Key |
|------|----------|----------|
| Collection | `build_collection_agent()` | `collection_result` |
| Ranking | `build_ranking_agent()` | `ranking_result` |
| Briefing | `build_briefing_agent()` | `briefing_result` |

### 5.5 observe_results（结果观察）

**位置**：`agents/main_agent.py:722`

**职责**：调用 `observe.evaluate` hook（默认实现 `_default_observe_evaluate`）评估子 Agent 结果。

**评估维度**：
| 维度 | 阈值 | 说明 |
|------|------|------|
| 采集量 | >= 3 条 | 采集是否充足 |
| 排序质量 | top_score >= 0.3 | 排序是否有效 |
| 简报质量 | >= 0.7 | 简报是否达标 |
| 简报条目 | >= 10 条 | 简报内容是否丰富 |

**特殊处理**：
- **预筛过严检测**：采集充足但排序后条目极少时，标记为 `expand_threshold` 而非排序失败
- **Briefing 内部重试上限**：如果 Briefing Agent 内部已达重试上限，接受当前结果不再触发 Planner 重试

### 5.6 coordinator_reflect（综合审查）

**位置**：`agents/main_agent.py:767`

**职责**：调用 `reflect.check` hook 进行多维质量审查：

1. **完整性检查**：排序条目是否为空
2. **简报质量检查**：评分是否过低
3. **去重检查**：高相似条目是否有标注
4. **矛盾检查**：简报条目间是否存在事实矛盾
5. **来源追溯检查**：条目是否有来源 URL
6. **循环次数检查**：ReAct 是否过多

### 5.7 push_notification（推送）

**位置**：`agents/main_agent.py:816`

**职责**：通过 MCP Client 推送简报。优先推送 Markdown 渲染版，失败时降级为 ranked_items 摘要。

### 5.8 update_memory（记忆更新）

**位置**：`agents/main_agent.py:887`

**职责**：管线收尾，完成以下操作：

1. **写入情节记忆**：通过 `add_memory()` 写入 `execution_logs`（SQLite）
2. **LLM 摘要写入长期记忆**：对本次执行做 LLM 摘要，写入 ChromaDB `domain_knowledge`
3. **写入 run_logs**：记录采集量、排序量、简报质量等
4. **保存简报**：
   - 写入 `raw_items` 和 `deduped_items`
   - 建立原始 ID → dedup_id 映射
   - 替换 briefing_data 中的 ID
   - 写入 `briefs` 表
   - 写入 `briefing_items` 关联表
5. **更新偏好向量**：将 top 3 条正向偏好写入 ChromaDB `user_preference`
6. **写入条目历史向量**：将 ranked_items 写入 ChromaDB `feed_items`（用于后续跨批次预过滤去重）

## 六、子 Agent 详解

### 6.1 Collection Agent（采集）

**位置**：`agents/collection_agent.py`

**模式**：默认为 **Pipeline（固定流水线）**，通过 `config.yaml` 中 `agents.collection_mode` 配置，可选 "react" 模式。

**Pipeline 执行流程**：
```
Step 1: fetch_rss（从 sources 表读取活跃源，并行抓取 RSS）
   │
   ├─ 向量预过滤（ChromaDB feed_items 去重，阈值 0.92）
   │
   ▼
Step 2: search_web（仅当 RSS 采集量 < collection_search_threshold 时触发）
   │
   ├─ 向量预过滤（同上）
   │
   ▼
Step 3: normalize_items（统一字段格式）
   │
   ▼
返回 collected_items
```

**ReAct 模式**：LLM 自主调用 `fetch_rss` → `search_web` → `normalize_items` → `finish_task`

**向量预过滤**（`_prefilter_against_history`）：
- 对每个条目用 bge-small 编码 title+summary
- 在 ChromaDB `feed_items` 集合中查最相似的历史条目
- 余弦相似度 >= 0.92 → 直接丢弃（历史已有）
- 任何异常都透传全部条目，不阻塞管线

### 6.2 Ranking Agent（排序）

**位置**：`agents/ranking_agent.py`

**ReAct 执行流程**：
```
第1轮：LLM 决策
   ├─ vector_search（查询用户偏好和反馈历史）
   ├─ deduplicate（向量去重 + LLM 裁决中间区间）
   │     ├─ 相似度 >= 0.88 → 自动判重
   │     ├─ 相似度 0.70~0.88 → LLM 裁决
   │     └─ 相似度 < 0.70 → 保留
   └─ 继续思考...

第2轮：LLM 决策
   └─ rank_items（多因子加权排序）
         ├─ 预筛（3天时间窗口，expand模式14天）
         ├─ 权重模式切换：
         │    ├─ 冷启动（feedback < 3）：similarity:0.4, recency:0.25, preference:0.1, importance:0.25
         │    └─ 有反馈（feedback >= 3）：similarity:0.3, recency:0.2, preference:0.4, importance:0.1
         ├─ 多因子计算：
         │    ├─ similarity：条目与 goal_embedding 的余弦相似度
         │    ├─ recency：指数衰减 exp(-hours/24)
         │    ├─ preference：like/dislike 向量偏好（正负分离）
         │    └─ importance：条目重要性评分
         ├─ 反馈偏置：like+0.15, dislike-0.10, irrelevant-0.15
         └─ 截断：top 10（expand模式 top 20）

第3轮：LLM 决策
   └─ finish_task（完成）
```

**排序公式**：
```
final_score = w_sim * similarity + w_rec * recency + w_pref * preference + w_imp * importance
```

### 6.3 Briefing Agent（简报生成）

**位置**：`agents/briefing_agent.py`

**ReAct 执行流程（P1-08 优化版）**：
```
第1轮：LLM 决策
   └─ generate_briefing
         ├─ 预检（preflight_for_briefing）：
         │    ├─ 过滤低质条目（_score < 0.15 或无标题）
         │    ├─ URL 去重合并
         │    ├─ 时间跨度检查
         │    └─ 条目数量检查
         ├─ 分组（_group_by_category）：按分类组织条目
         ├─ LLM 生成结构化 JSON 简报
         ├─ 回填（_backfill_briefing_items）：补全缺失字段
         ├─ Markdown 渲染（_render_markdown）
         └─ 代码层自动质量评估（brief_quality_check_node）：
              ├─ completeness（完整性）：条目覆盖率
              ├─ relevance（相关性）：LLM 独立评估（首次计算后缓存复用）
              └─ coherence（一致性）：矛盾检测
              综合评分 = completeness×0.3 + relevance×0.4 + coherence×0.3

如果质量达标（>= 0.7）→ 自动完成
如果质量不达标 → 注入质量反馈，自动重试（最多2次）
达到重试上限 → 强制收敛
```

## 七、反馈系统（Feedback Agent）

**位置**：`agents/feedback_agent.py`

### 7.1 触发方式

用户在首页简报中点击 👍/👎/🚫 按钮时，UI 调用 `process_feedback_async()`，**异步**执行反馈处理，不阻塞主 Agent。

### 7.2 执行流程

```
record_feedback → update_preference → vector_add → cleanup_preference → END
```

**1. record_feedback**：写入 SQLite `feedback` 表
**2. update_preference**：
   - 查询条目信息（从 `deduped_items` + `raw_items`）
   - 提取关键词（从条目 keywords 或标题摘要分词）
   - **EMA 更新偏好向量**：`v_new = α * v_current + (1-α) * v_feedback`（α=0.3）
   - 更新 SQLite `user_preferences` 表（关键词级别）
**3. vector_add**：将更新后的 v_like/v_dislike 写回 ChromaDB `user_preference`
**4. cleanup_preference**：移除权重 < 0.1 的低权重偏好项

### 7.3 偏好权重算法

```python
FEEDBACK_BIAS = {"like": 0.15, "dislike": -0.10, "irrelevant": -0.15}
# 反馈次数越多，权重绝对值越大
magnitude = min(count * 0.15, 0.8)
# like: base_bias + magnitude（上限 1.0）
# dislike/irrelevant: base_bias - magnitude（下限 -1.0）
```

## 八、记忆系统

**位置**：`utils/memory_manager.py`

### 8.1 双层架构

| 层级 | 存储 | 用途 | 检索方式 |
|------|------|------|----------|
| 情节记忆 | SQLite `execution_logs` | 记录每次执行的决策和结果 | 按时间范围查询（近7天） |
| 长期记忆 | ChromaDB `domain_knowledge` | LLM 摘要后的历史经验 | 语义向量检索 |

### 8.2 写入时机

在 `update_memory_node` 中触发：
1. 写入情节记忆（`execution_logs` 表）
2. LLM 对本次执行做摘要（200字以内），写入 ChromaDB

### 8.3 检索时机

在 `planner_node` 中触发：
1. 检索近7天情节记忆（最多10条），帮助 Planner 回顾近期执行效果
2. 检索语义相似的长期记忆（最多3条），获取历史类似场景的处理经验

## 十一、关键设计决策

### 11.1 ReAct 循环

- Planner 输出编排计划 → 子 Agent 执行 → Observe 评估 → Planner 再思考
- 最多 3 轮 ReAct 循环（`react_cycle_count`）
- 第 3 轮强制收敛，不重新编排

### 11.2 规则优先路由

- 正常流程中所有路由场景均由规则判断（确定性），不调用 LLM
- 只有在需要重新编排 `sub_agent_plan` 时才路由到 planner（需要 LLM）
- 大幅减少 LLM 调用次数和路由不确定性

### 11.3 错误隔离

- 子 Agent 通过 `run_with_isolation` 隔离执行
- 单个子 Agent 失败不阻断本轮其余子 Agent
- 向量预过滤、记忆写入等非关键操作降级透传，不阻塞主流程

### 11.4 冷启动 vs 有反馈模式

- 冷启动（feedback < 3）：偏好因子权重仅 0.10，主要依赖相似度和时效性
- 有反馈（feedback >= 3）：偏好因子权重升至 0.40，个性化程度显著提升
- 反馈系统通过 EMA 算法持续更新用户偏好向量

### 11.5 Collection Agent 的双模式

- **Pipeline 模式**（默认）：固定流水线，无 LLM 参与，节省 API 成本
- **ReAct 模式**：LLM 自主决策，灵活但消耗更多 token

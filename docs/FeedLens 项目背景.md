## 一、行业背景与痛点

### 1.1 信息过载问题

在信息爆炸的时代，用户面临的核心问题已从「信息匮乏」转变为「信息过载」。根据行业研究，普通知识工作者每天接触的信息量约为 34GB，但有效利用率不足 5%。主要原因包括：

- **来源分散**：社交媒体、新闻网站、行业博客、RSS 订阅源等超过数十个渠道
- **重复信息**：同一事件在不同平台被多次报道，用户被迫阅读大量冗余内容
- **质量参差**：缺乏有效筛选机制，高价值内容被淹没在噪音中
- **时间消耗**：手动筛选和阅读需要大量时间，影响核心工作效率

### 1.2 现有工具的局限性

| 工具类型 | 代表产品 | 核心局限 |
|---------|---------|---------|
| 传统 RSS 阅读器 | Feedly、Inoreader | 被动拉取、无个性化筛选、按时间排序 |
| 新闻聚合应用 | 今日头条、腾讯新闻 | 基于关键词的硬匹配、无法去重、推荐算法不透明 |
| AI 问答工具 | ChatGPT、Claude | 被动响应、无定时推送、无持续学习能力 |
| 定时脚本 + Pipeline | Cron + Python 脚本 | 固定流程执行、无法自主决策、无法适应反馈 |

### 1.3 市场机会

用户存在一个未被很好满足的需求：**希望每天收到一份经过智能筛选、去重、排序的高质量信息简报，而无需手动操作**。这个需求的特点是：

- **主动交付**：不是用户主动查看，而是系统定时推送
- **高度个性化**：基于用户反馈持续学习偏好
- **质量优先**：不是数量堆砌，而是精心筛选的精华
- **持续优化**：反馈闭环让系统越用越懂用户

## 二、项目愿景与核心价值

### 2.1 项目定位

FeedLens 是一个**主动式信息聚合 Agent 系统**。它不是被动问答工具，也不是 cron + pipeline，而是能**自主规划、调度子 Agent、定时执行、个性化筛选**的多 Agent 智能体系统。

### 2.2 核心差异化

| 维度 | 传统工具 | FeedLens |
|------|---------|----------|
| **信息获取** | 被动拉取，用户自行浏览 | 定时自主采集 + 搜索补充 |
| **信息过滤** | 按时间排序，用户自行筛选 | 智能去重 + 偏好排序，推送高价值内容 |
| **个性化** | 基于关键词的硬匹配 | 基于用户反馈动态学习偏好向量 |
| **输出形态** | 原始条目列表 | 结构化简报（分类 + 重要性标注 + 来源引用） |
| **交互方式** | 用户主动查看 | Agent 定时推送 + 用户反馈闭环 |
| **决策方式** | 硬编码流程 | **Planner 自主编排**，根据状态动态决策 |

### 2.3 MVP 核心假设

> **假设**：用户愿意每天收到一份「5-10 条高价值、已去重、按个人偏好排序」的信息简报，并通过反馈持续改善推送质量。

MVP 阶段的一切设计决策均围绕**最快验证此假设**展开。

### 2.4 MVP 交付范围

**做什么**：
1. 主 Agent planner 自主编排子 Agent（采集 / 排序 / 简报）
2. 多因子偏好排序 + 反馈闭环个性化
3. APScheduler 定时触发 + 重大事件破例推送
4. Streamlit 应用内交付（简报 + 反馈 + 配置）

**不做什么**：多用户认证、Telegram推送、Docker容器化、跨类别配额、简报风格切换。

## 三、技术选型背景

### 3.1 选型原则

MVP 阶段的技术选型遵循三个核心原则：

| 原则 | 含义 |
|------|------|
| **快速验证** | 优先选择能快速实现和迭代的技术，不追求完美 |
| **本地优先** | 避免云服务依赖，减少部署复杂度 |
| **自主可控** | 核心能力不依赖第三方服务，保证稳定性和成本可控 |

### 3.2 关键技术选型理由

| 技术领域 | 选择方案 | 选型理由 |
|---------|---------|---------|
| **Agent 编排框架** | LangGraph StateGraph | 原生支持 ReAct 循环、子图、状态传递；9/9 设计文档共识，零争议 |
| **LLM** | DeepSeek Chat | 性价比高，支持 Function Calling，本地运行无速率限制 |
| **Embedding 模型** | bge-small-zh-v1.5 | 轻量级（384 维）、中文支持好、推理速度 < 100ms/条、本地运行免费 |
| **向量数据库** | ChromaDB | 轻量、易用、支持元数据过滤，本地文件存储无需部署 |
| **关系型数据库** | SQLite + WAL 模式 | 轻量、无需部署、支持并发读写 |
| **定时任务** | APScheduler | Python 原生、灵活、支持 cron，与主流程无缝集成 |
| **用户界面** | Streamlit | 快速构建数据应用，Python 原生，学习成本低 |

### 3.3 为什么选择多 Agent 架构

多 Agent 架构是本项目的核心差异化选择，原因如下：

1. **Planner 自主编排能力**：这是展示 Agent 智能的关键——Planner 能根据当前状态自主决定调用哪些子 Agent、什么顺序、是否需要补充数据，而非硬编码执行
2. **ReAct 循环支持**：每个子 Agent 都能独立运行 Think→Act→Observe 循环，实现真正的自主决策
3. **模块化与可维护性**：采集、排序、简报、反馈四个职责分离，便于独立迭代
4. **错误隔离**：单个 Agent 失败不会导致整体崩溃

### 3.4 为什么不是单 Agent

| 方案 | 问题 |
|------|------|
| 单 Agent + 增强 Planner | Planner 放在 P1 增强，MVP 交付时没有展示核心价值 |
| 单 Agent + ReAct 仅在关键节点 | 只在 collect_sources 和 rank_items 有 ReAct，不够自主 |
| 纯 Python asyncio | 无编排框架支持，需手动实现状态管理、条件路由、重试逻辑 |

## 四、架构设计哲学

### 4.1 六层架构

```
展示层 (Streamlit) → 规划层 (主Agent) → 执行层 (子Agent) → 工具层 (FC/MCP) → 记忆层 → 数据层
```

- **展示层**：用户交互入口，简报呈现 + 反馈收集
- **规划层**：主 Agent 的 Planner，统一决策调度
- **执行层**：子 Agent 各司其职，采集/排序/简报/反馈
- **工具层**：14 个 FC 工具 + MCP 服务，提供原子能力
- **记忆层**：情节记忆（SQLite，近7天）+ 长期记忆（ChromaDB，语义检索）
- **数据层**：SQLite 存储结构化数据，ChromaDB 存储向量

### 4.2 多 Agent 职责分工

| Agent | 职责 | 自主能力 |
|-------|------|---------|
| **主 Agent** | 自主编排子 Agent、综合质量审查、推送交付 | ✅ Planner 自主决策 + ReAct 循环 |
| **采集 Agent** | RSS 采集 + 搜索补充 + 元数据提取 | ✅ Pipeline/ReAct 双模式（默认 Pipeline） |
| **排序 Agent** | 智能去重 + 偏好排序 | ✅ Think→Act→Observe 循环 |
| **简报 Agent** | 简报生成 + 质量审查 | ✅ ReAct 循环 + 代码层自动质量评估 |
| **反馈 Agent** | 反馈处理 + 偏好向量更新（异步） | ❌ 单次执行 |

### 4.3 业务闭环

```
用户设定 Goal → LLM 提取偏好 → APScheduler 定时触发
       ↓
主 Agent planner 自主编排子 Agent
       ↓
采集 Agent → 排序 Agent → 简报 Agent → reflect
       ↓
推送（定时 + 重大事件破例）→ 用户反馈
       ↓
反馈子 Agent → 偏好向量更新 → 影响后续排序
```

## 一、冲突的共识点分析

> 本節分析：哪些冲突点的「对立双方」实际上共享了某个更底层的共识？
> 即：冲突是「怎么做」的分歧，但「做什么」是共识。

| 冲突编号 | 冲突事项 | 底层共享共识 | 共识如何限制冲突的可选范围 |
|---------|---------|-------------|---------------------------|
| P1 | 简报生成驱动模式 | **共识：核心工作流为采集→去重→排序→生成→推送** | 无论流程驱动还是 Goal 驱动，这 5 个节点必须都存在，分歧仅在「是否由 Planner 动态决定执行顺序」 |
| P2 | 推送机制 | **共识：APScheduler 定时调度 + Streamlit 前端展示** | 自主推送只能作为「定时推送」的补充（破例触发），不能替代调度基础设施 |
| P3 | 用户配置粒度 | **共识：用户反馈驱动偏好学习** | Goal 文本输入后，仍需用 LLM 从 Goal 中提取结构化字段（topics/keywords/sources），这实际上是「多维度配置」的自动版，而非真正的替代 |
| P4 | 去重结果处理 | **共识：向量去重 + 同事件不同角度区分** | 无论折叠分组还是计数标注，都必须先有「同事件不同角度」的识别能力，分歧仅在「前端展示形式」 |
| P5 | 搜索采集角色 | **共识：RSS 解析用 feedparser + 搜索 API 补充** | RSS 是主通道（免费、稳定），搜索是补充（成本、速率限制），冲突双方都不反对这个优先级 |
| P6 | 简报输出风格 | **共识：LLM 生成简报 + 含重要性标注和来源引用** | JSON 中间格式还是直接 Markdown，是工程实现分歧，不影响产品输出 |
| P7 | 用户反馈选项 | **共识：反馈影响偏好向量，进而影响排序** | 无论二元还是多元，反馈信号必须可量化为偏好向量更新，这是硬约束 |
| T1 | MCP Push 部署模式 | **共识：MCP 协议用于外部服务封装** | stdio 还是 SSE，取决于「推送服务是否需跨进程/跨机器」，MVP 阶段无此需求，stdio 更简 |
| T2 | 向量去重阈值 | **共识：去重需阈值 + 模糊区间需 LLM 裁决** | 阈值具体数值是经验问题，共识是「需要校准脚本」，而非「阈值必须是 X」 |
| T3 | Embedding 模型选择 | **共识：用中文优化的 sentence-transformers 本地模型** | API vs 本地是成本分歧，但「中文优化」是共识（无一文档选用纯英文模型） |
| T4 | MCP Server 数量 | **共识：混合工具调用策略（FC + MCP）** | 分歧仅在「DB/向量操作是否值得封装为 MCP」，FC/MCP 边界划分原则是共识 |
| T5 | 排序权重配置 | **共识：多因子加权排序公式** | 权重数值是产品调参分歧，但「四因子框架」是共识（similarity/recency/preference/authority） |
| T6 | 去重技术路线 | **共识：向量相似度是主干 + 需处理模糊区间** | 纯向量、向量+编辑距离、向量+NER，都是「向量主干 + 辅助信号」的变体 |
| T7 | 短期记忆窗口 | **共识：滑动窗口 + 超窗时压缩写入情节记忆** | 10 轮 vs 15 轮是数值分歧，但「窗口+压缩」机制是共识（Kimi/Codex_Mimo 均提及） |
| T8 | 时间衰减函数 | **共识：指数衰减** | 数学表达式差异（λ vs τ）实质等价，这是两份整合版均未明确指出的重要共识 |
| T9 | 反思重试次数 | **共识：反思节点 + 不合格触发重试** | 1 次 vs 2 次是数值分歧，「必须有重试机制」是共识 |
| T10 | 向量库交互方式 | **共识：ChromaDB 做向量存储** | FC 直调 vs MCP 解耦，是架构分歧，但「用 ChromaDB」是共识 |
| T11 | 部署方案 | **共识：MVP 阶段优先跑通功能** | 本地运行 vs Docker Compose，是交付形式分歧，但「MVP 先跑通」是共识 |

**关键洞察**：18 个冲突点中，有 **0 个冲突是「要不要做」的分歧，全部是「怎么做」的分歧**。这意味着 9 份原始文档在设计愿景上高度一致，分歧全在技术选型和参数调优层面。这对 MVP 决策是极有利的信号——不需要重新设计工作流，只需在已有框架上做选择题。

- **来源依据**：GPT 文档第三节 / Codex_Mimo 2.1 节 / 全部文档 Agent 工作流设计章节
- **两份差异**：WB 明确标注「9 份」，TRAE 未标数量
- **MVP 约束力**：★★★★★（硬约束——MVP 技术选型必须基于 LangGraph，不可更换为其他框架）

---

**共识 2：四/五层分层架构**

- **内容**：全部文档均将系统划分为感知层、大脑层（LLM）、工具层、记忆层、规划层。Codex_Mimo 提供了最完整的六层架构映射表（含展示层）。
- **来源依据**：全部文档 · 系统架构图章节

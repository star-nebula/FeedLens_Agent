### 一、项目定位

**FeedLens 是一个基于 LangGraph 的智能信息简报 Agent** —— 通过 LLM 驱动的 ReAct 循环与反思机制，自动从 RSS 和搜索引擎采集用户关注领域的最新信息，经去重、排序、摘要后生成结构化简报，并通过反馈闭环持续学习用户偏好。

### 四、独特点提取

- **亮点1：Goal 驱动的自主 Agent 设计**：来自 **GPT文档-核心Goal章 / 自主推送机制章**。该设计让 Agent 围绕用户长期目标自主决策“是否搜索、是否继续、是否立即推送”，而非执行固定日报流程。价值：体现 Agent 的自主规划能力，是简历中“Agent 决策”而非“流水线”的差异化亮点。

- **亮点2：MCP 双 Transport 对比实现**：来自 **GLM文档-工具清单章**。刻意让 web_search 使用 SSE、push_notifier 使用 stdio，展示对 MCP 协议两种部署形态的理解。价值：体现对 MCP 协议的深度掌握，可在简历中写出“支持 SSE 和 stdio 双模式 MCP Server”。

- **亮点3：向量粗排 + LLM 精排两阶段去重**：来自 **Qwen文档-去重策略章 / Perplexity文档-去重策略章**。先用低成本向量阈值（0.70-0.88）粗筛，再用 LLM 对模糊地带做事件级判别，区分“真重复”与“同事件不同角度”。价值：平衡去重精度与成本，比纯阈值方案更工程化。

- **亮点4：带可复现校准流程的去重阈值**：来自 **GLM文档-去重策略章**。设计了“人工标注 200 对样本 → 扫描阈值区间 → 绘制 P/R/F1 曲线 → 选最优阈值”的校准脚本。价值：体现数据驱动的工程思维，简历可写“带可复现阈值校准流程”。

- **亮点5：权重动态调整机制（在线学习）**：来自 **TRAE文档-排序算法章**。根据用户连续反馈自动微调排序权重（如连续 3 次 like 则 +0.05 上限 0.5），并做权重归一化。价值：比静态权重更体现“持续学习”，是简历中“自适应排序”的技术亮点。

- **亮点6：跨类别配额机制**：来自 **GLM文档-排序算法章 / TRAE文档-排序算法章**。排序后按用户话题数均分配额（如 5 话题 × 4 条 = 20），避免单一热门话题霸屏。价值：提升简报内容多样性，体现产品体验意识。

- **亮点7：执行日志（Execution Logs）作为情节记忆的结构化存储**：来自 **Codex_DeepSeek文档-数据模型章**。设计了 execution_logs 表记录每个节点的运行状态、耗时、处理条目数。价值：为后续“情节记忆反思”和“执行仪表盘”提供结构化数据基础。

- **亮点8：RSS 与搜索的并行采集 + 条件合并**：来自 **Kimi文档-工作流图章 / TRAE文档-条件分支章**。RSS 和搜索并行执行，若 RSS 条目不足则触发搜索补充；搜索失败时允许换关键词重试。价值：体现容错设计和采集策略的鲁棒性。

- **亮点9：简报质量的结构化评分体系**：来自 **Kimi文档-2.2节 State 定义 / reflect_quality 工具**。在 State 中定义 brief_quality 字段（含 completeness、relevance、coherence、score 四个维度），质量分 < 0.7 触发重试。价值：质量评估可量化、可追踪，比“是否通过”的二元判断更精细。

- **亮点10：情节记忆的向量化检索**：来自 **GLM文档-记忆系统章**。不仅将情节记忆存入 SQLite，还将其摘要向量化存入 ChromaDB，支持“相似执行经验检索”。价值：让 Agent 可以从历史成功/失败经验中学习，比纯 SQL 查询更智能。

- **亮点11：用户偏好关键词表 + 权重累积**：来自 **Codex_Mimo文档-数据模型章 / Kimi文档-数据模型章**。设计了 user_preference_keywords 表存储用户偏好的具体关键词及其权重，权重范围 [0,1]，低于阈值自动清理。价值：偏好表达可解释、可调试，便于人工审查和手动修正。

- **亮点12：去重中的“同事件不同角度”保留策略**：来自 **Codex_DeepSeek文档-去重策略章 / Codex_Mimo文档-去重策略章**。对相似度 [0.65, 0.85) 的条目标记为 related（同事件不同角度），合并为多源条目并给予 source_diversity_bonus +0.05。价值：兼顾去重与信息多样性，体现产品细节思考。

- **亮点13：Docker Compose 一键部署**：来自 **GLM文档-阶段6章 / Perplexity文档-阶段6章**。将 Agent、Streamlit、ChromaDB、MCP Server、Searxng 全部容器化，支持 `docker compose up` 一键启动。价值：大幅降低项目演示门槛，简历可写“容器化部署”。

- **亮点14：情节记忆的“执行仪表盘”**：来自 **GLM文档-阶段5章 / Codex_DeepSeek文档-P5章**。Streamlit 页面展示执行成功率、耗时、去重率、反馈率等历史指标。价值：可视化 Agent 的运行效果，增强演示说服力。

- **亮点15：低代码 Prompt 模板设计**：来自 **GPT文档-Planner Prompt 章**。Planner 的 Prompt 直接输出 JSON 格式的 action，包含 Search / SearchMore / GenerateBrief / PushNow / Stop 五种决策。价值：极简的 Planner 设计，便于快速迭代和调试。

### 五、总结：设计决策图谱

| 维度 | 主流方案（7+ 文档） | 少数方案（1-2 文档） |
|------|-------------------|-------------------|
| 驱动模式 | 流程驱动（定时执行） | Goal 驱动（自主决策） |
| 推送策略 | 定时推送 / Streamlit 展示 | 基于重要性自主推送 |
| 用户配置 | 多维度结构化配置 | 单 Goal 文本配置 |
| MCP Search | SSE 部署 | stdio 部署 |
| Embedding | 本地 sentence-transformers | DeepSeek Embedding API |
| 去重阶段 | 两阶段（向量 + 规则） | 三阶段（+ LLM 精排） |
| 排序核心 | 偏好优先（w=0.30-0.45） | 相关性优先（w=0.40） |
| 去重阈值 | 0.85-0.88 | 0.90-0.95 |
| MCP 数量 | 2 个（search + push） | 3-4 个（+ db + vector） |
| 短期记忆窗口 | 10 轮 | 15 轮 |
| 反思重试次数 | 2-3 次 | 1 次 |

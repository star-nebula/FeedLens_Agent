# FeedLens 系统整体架构图（v3.0 多Agent自主规划）

> 本文档替代 `系统整体架构图.drawio`，以结构化 Markdown 彏述 FeedLens 系统的六层架构。

---

## 架构总览

FeedLens 采用 **六层垂直架构**，从感知层（数据输入）到展示层（用户交互），中间由规划层自主编排各子Agent完成任务。核心特征是 **planner 自主决策**而非线性流水线。

```
┌─────────────────────────────────────────────────────────────────────┐
│  展示层 (Streamlit)                                                  │
│  配置界面 | 简报阅读 | 反馈操作 | 执行仪表盘                           │
├─────────────────────────────────────────────────────────────────────┤
│  规划层 — 主Agent (Coordinator + Planner)                            │
│  understand_intent → planner(P0) → invoke_sub_agent → observe       │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │ 子Agent（由 planner 自主调度）                                │    │
│  │ 采集Agent(ReAct) | 排序Agent(ReAct) | 简报Agent(线性)        │    │
│  └─────────────────────────────────────────────────────────────┘    │
│  反馈Agent(异步) ← 用户反馈触发                                      │
│  reflect → push_notification → update_memory                        │
├─────────────────────────────────────────────────────────────────────┤
│  大脑层 (LLM)                                                        │
│  DeepSeek（主力） | LLMProvider 抽象接口（预留双供应商扩展点）          │
├─────────────────────────────────────────────────────────────────────┤
│  工具层 (FC + MCP + 子Agent)                                         │
│  FC工具(10+,进程内) | MCP Server(2,独立进程) | 子Agent(4,StateGraph)  │
├─────────────────────────────────────────────────────────────────────┤
│  记忆层                                                               │
│  短期(15轮滑动窗口) | 长期(ChromaDB偏好) | 语义(ChromaDB条目)         │
│  情节(SQLite执行日志) | 超窗压缩(LLM→ChromaDB)                       │
├─────────────────────────────────────────────────────────────────────┤
│  感知层                                                               │
│  RSS输入 | 搜索结果 | 用户Goal | 用户反馈                             │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 第6层：展示层（Streamlit）

| 组件 | 功能 | 对应主流程节点 |
|------|------|---------------|
| 配置界面 | RSS源管理、偏好Goal设置、推送时间配置 | `understand_intent` 的输入源 |
| 简报阅读 | Markdown渲染、分类分组、计数标注显示 | `push_notification` 的输出 |
| 反馈操作 | like/dislike/irrelevant 三级反馈 | `FeedbackAgent` 的触发源 |
| 执行仪表盘 | 运行状态、采集数/去重率/质量分数 | `execution_logs` 的可视化 |

**交互方式**：Streamlit 单页应用，APScheduler 后台触发 + 用户手动触发双通道。

## 第5层：规划层 — 主Agent（Coordinator + Planner）

这是系统的**核心决策层**，决定整个架构从"流水线Agent"升级为"自主规划Agent"。

### 主Agent节点序列

| 节点 | 职责 | ReAct步骤 | 输出 |
|------|------|-----------|------|
| `understand_intent` | 识别触发类型，提取结构化偏好 | 无 | `trigger_type`, `structured_goal` |
| **`planner` (P0核心)** | LLM自主编排子Agent调用计划 | **Think** | `sub_agent_plan`, `planner_reasoning` |
| `invoke_sub_agent` | 按plan调度子Agent | **Act** | `sub_agent_results` |
| `observe_results` | 评估子Agent输出质量 | **Observe** | 观察评估结论 |
| `reflect` | 综合质量审查、矛盾检查 | — | `brief_quality`, `reflection_notes` |
| `push_notification` | MCP stdio推送 | — | 推送成功/失败 |
| `update_memory` | 偏好向量更新、执行日志写入 | — | 更新完成状态 |

### ReAct循环（planner核心闭环）

```
Think (planner) → Act (invoke_sub_agent) → Observe (observe_results)
    ↑                                              │
    └────── 需要补充? ──────────────────────────────┘
```

- **最多3个ReAct循环**，防止无限循环
- planner 每次循环决定：调用哪个子Agent、传什么参数、是否跳过某步骤
- 循环结束条件：planner 判断 "Done" 或达到最大循环数

### 子Agent框（由 planner 自主调度）

| 子Agent | 模式 | 核心工具 | 输出 |
|---------|------|---------|------|
| 采集Agent | **ReAct循环** | fetch_rss(FC) + search_web(MCP SSE) | `normalized_items` |
| 排序Agent | **ReAct循环** | deduplicate(FC) + rank_items(FC) + vector_search(FC) | `ranked_items` |
| 简报Agent | **线性流程** | generate_briefing(FC) + reflect(FC) | `briefing`, `brief_quality` |
| 反馈Agent | **异步触发** | update_preference(FC) + vector_add(FC) | 偏好向量更新 |

### 反馈Agent（异步）

- **独立运行**，不阻塞主流程
- 由用户反馈（like/dislike/irrelevant）直接触发
- 偏好向量更新后影响后续排序Agent的 preference 因子权重

| 配置项 | 值 |
|--------|---|
| 主力模型 | DeepSeek |

## 第3层：工具层（FC + MCP + 子Agent）

### 三类工具对比

| 工具类型 | 调用方式 | IPC开销 | 部署方式 | 数量 |
|----------|---------|---------|---------|------|
| **FC (Function Calling)** | 进程内直接调用 | 无 | 无需独立部署 | 10+ |
| **MCP Server** | HTTP SSE / stdio pipe | 有 | 独立进程 | 2 |
| **子Agent** | LangGraph子图调用 | 低 | 同进程内StateGraph | 4 |

### FC工具清单

| 工具名 | 所属Agent | 依赖 | 返回类型 |
|--------|----------|------|---------|
| `fetch_rss` | 采集Agent | feedparser | `list[dict]` |
| `enrich_metadata` | 采集Agent | LLM单次推理 | `list[dict]` |
| `normalize_items` | 采集Agent | 无(纯数据转换) | `list[dict]` |
| `deduplicate` | 排序Agent | ChromaDB SDK | `list + 关系` |
| `rank_items` | 排序Agent | 无(纯计算) | `list[dict]` |
| `vector_search` | 排序Agent | ChromaDB SDK | `list[dict]` |
| `generate_briefing` | 简报Agent | LLM单次调用 | `dict` |
| `reflect` | 简报Agent | LLM单次调用 | `dict` |
| `update_preference` | 反馈Agent | EMA计算 | `dict` |
| `vector_add` | 反馈Agent | ChromaDB SDK | `list[str]` |
| `db_read` | 共享 | SQLite | `list[dict]` |
| `db_write` | 共享 | SQLite | `bool` |

### MCP Server清单

| MCP Server | 传输模式 | 端口 | 部署方式 | 方法 |
|-------------|---------|------|---------|------|
| `search_web` | **SSE** | :8100 | 独立进程常驻 | `search(query, max_results)` |
| `push_notification` | **stdio** | 无 | 子进程随主进程启停 | `push(brief, user_id, immediate)` |

## 第2层：记忆层

### 四层记忆架构

| 记忆类型 | 存储 | 内容 | 生命周期 |
|----------|------|------|---------|
| **短期记忆** | State 滑动窗口 | 最近15轮对话/操作上下文 | 单次Session |
| **长期记忆** | ChromaDB `user_preference` | 用户偏好向量(v_like/v_dislike) | 跨Session持久 |
| **语义记忆** | ChromaDB `feed_items` | 条目embedding向量 | 跨Session持久 |
| **情节记忆** | SQLite `execution_logs` / `run_logs` | 执行日志、运行指标 | 跨Session持久 |

### 超窗压缩机制

- 短期记忆窗口满15轮时，通过LLM压缩为摘要
- 压缩结果写入 ChromaDB 长期记忆
- 保证上下文不丢失，同时控制State体积

## 层间交互关系

| 交互路径 | 方向 | 说明 |
|----------|------|------|
| 展示层 → 规划层 | 下行 | 用户配置/Goal/反馈 → understand_intent |
| 规划层 → 大脑层 | 下行 | planner Think + 各子Agent LLM推理 |
| 规划层 → 工具层 | 下行 | invoke_sub_agent → FC/MCP/子Agent |
| 工具层 → 记忆层 | 下行 | db_read/db_write/vector_search/vector_add |
| 记忆层 → 感知层 | 下行 | RSS/搜索原始数据存入 feed_items |
| 规划层 → 展示层 | 上行 | push_notification → Streamlit推送 |
| 反馈 → 规划层 | 上行 | 用户反馈 → FeedbackAgent → 偏好更新 → 影响排序 |

## 架构关键设计点

1. **planner 在P0而非P1** — 自主决策是核心闭环，不是增强功能
2. **子Agent作为工具注册** — 对应 Prompt 中 "skills、MCP、子Agent" 三种工具类型
3. **ReAct循环只在需要自主决策的地方** — 采集Agent和排序Agent有ReAct，简报Agent线性执行
4. **反馈Agent异步** — 不阻塞主流程，偏好更新通过向量影响后续排序
5. **MCP双传输** — search_web用SSE（外部HTTP长连接），push_notification用stdio（内部推送短连接）
6. **记忆四层分离** — 短期/长期/语义/情节各有独立存储，超窗压缩防止信息丢失

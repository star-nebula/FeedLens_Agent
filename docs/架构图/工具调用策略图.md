# FeedLens 工具调用策略图（v3.0 三层工具架构）

> 本文档替代 `工具调用策略图.drawio`，以结构化 Markdown 描述子Agent / MCP / FC 三层工具的注册、分类、调用链路和选型原则。

---

## 一、三层工具架构总览

FeedLens 的工具层对应 Prompt 中明确要求的 **"skills、MCP、子Agent"** 三种工具注册类型，形成三层混合策略：

```
┌─────────────────────────────────────────────────────────────────┐
│ 主 Agent (Coordinator + Planner)                                │
│                                                                 │
│ planner 输出编排计划:                                            │
│   sub_agent_plan = [                                             │
│     {agent: "Collection", params: {...}},                       │
│     {agent: "Ranking", params: {...}},                          │
│     {agent: "Briefing", params: {...}},                         │
│     skip: [],                                                    │
│     reason: str                                                  │
│   ]                                                              │
│                                                                 │
│ invoke_sub_agent → 选择工具类型 → 调度                            │
│     │                                                            │
│     ├── 子Agent (LangGraph子图调用)                              │
│     ├── MCP Server (IPC: SSE HTTP / stdio pipe)                │
│     └── FC (进程内直接调用, 无IPC)                                │
└─────────────────────────────────────────────────────────────────┘
```

---

## 二、第一层：子Agent — 独立StateGraph + 自主决策能力

### 2.1 采集Agent（CollectionAgent）

| 属性 | 值 |
|------|---|
| 模式 | **ReAct循环** |
| 调用方式 | LangGraph子图调用 (`add_node`) |
| 输入 | `{goal: dict, search_count: int}` |
| 输出 | `{raw_items: list, normalized_items: list, items_count: int}` |

**内部工具链**：

| 工具 | 类型 | 依赖 | 调用位置 | 说明 |
|------|------|------|---------|------|
| fetch_rss | FC | feedparser | Act步骤 | 并行采集多个RSS源，无状态 |
| search_web | **MCP SSE :8100** | 外部HTTP API | Act步骤（条件触发） | 需补充搜索时调用 |
| enrich_metadata | FC | LLM单次推理 | Act→Observe后 | 提取分类/关键词/重要性 |
| normalize_items | FC | 纯数据转换 | enrich后 | 统一字段格式 |

**ReAct循环**：
- Think → 判断是否需要搜索补充
- Observe → 评估采集结果（items < 5 → 补充搜索）

---

### 2.2 排序Agent（RankingAgent）

| 属性 | 值 |
|------|---|
| 模式 | **ReAct循环** |
| 调用方式 | LangGraph子图调用 (`add_node`) |
| 输入 | `{normalized_items: list, goal: dict, feedback_count: int}` |
| 输出 | `{deduped_items: list, ranked_items: list, dedup_rate: float}` |

**内部工具链**：

| 工具 | 类型 | 依赖 | 调用位置 | 说明 |
|------|------|------|---------|------|
| vector_search | FC | ChromaDB SDK | Think→Act | 检索偏好向量 |
| deduplicate | FC | ChromaDB SDK | Act步骤 | ≥0.88重复/0.70-0.88 LLM裁决/≤0.70不重复 |
| rank_items | FC | 纯计算 | Act步骤 | 多因子动态权重排序 |
| db_read | FC | SQLite | Think步骤 | 读取反馈历史 |

**ReAct循环**：
- Think → 检索偏好向量、规划排序策略
- Observe → 评估排序结果（偏好匹配度低 → 调参重排）

---

### 2.3 简报Agent（BriefingAgent）

| 属性 | 值 |
|------|---|
| 模式 | **线性流程**（无ReAct） |
| 调用方式 | LangGraph子图调用 (`add_node`) |
| 输入 | `{ranked_items: list, goal: dict}` |
| 输出 | `{briefing: dict, brief_quality: dict}` |

**内部工具链**：

| 工具 | 类型 | 依赖 | 说明 |
|------|------|------|------|
| generate_briefing | FC | LLM→JSON | 合并摘要+简报生成 |
| reflect | FC | LLM | 质量评分(completeness/relevance/coherence)，score<0.7重试 |

---

### 2.4 反馈Agent（FeedbackAgent）

| 属性 | 值 |
|------|---|
| 模式 | **异步触发**（不经过planner） |
| 调用方式 | LangGraph子图调用（独立入口） |
| 输入 | `{user_id, item_id, feedback_type}` |
| 输出 | 偏好向量更新完成 |

**内部工具链**：

| 工具 | 类型 | 依赖 | 说明 |
|------|------|------|------|
| update_preference | FC | EMA计算 | 偏好权重更新（正负分离） |
| vector_add | FC | ChromaDB SDK | 偏好向量写入 |

---

## 三、第二层：MCP Server — 需独立部署、涉及外部系统

### 3.1 search_web MCP Server

| 属性 | 值 |
|------|---|
| 传输协议 | **SSE** |
| 监听端口 | **:8100** |
| 部署模式 | 独立进程常驻运行 |
| 生命周期 | 常驻运行 |

**方法定义**：

```python
# MCP SSE 方法
def search(query: str, max_results: int = 10) -> list[dict]:
    """搜索外部HTTP API，返回结果列表"""
    pass
```

**调用链路**：
```
采集Agent → Think(需补充搜索)
  → Act(search_web)
    → MCP SSE HTTP请求 :8100
      → 搜索API返回结果
  → Observe(评估搜索结果)
```

**选型理由**：
- 搜索API是外部HTTP服务，需独立进程管理连接池
- SSE支持流式返回结果
- 长连接场景适合SSE模式
- 需端口管理(:8100)

---

### 3.2 push_notification MCP Server

| 属性 | 值 |
|------|---|
| 传输协议 | **stdio** |
| 部署模式 | 子进程随主进程启停 |
| 生命周期 | 随主进程启停 |
| 端口 | 无端口管理 |

**方法定义**：

```python
# MCP stdio 方法
def push(brief: dict, user_id: int, immediate: bool = False) -> bool:
    """推送简报给用户，immediate=True时破例推送"""
    pass
```

**调用链路**：
```
主Agent → reflect(质量合格)
  → push_notification
    → MCP stdio 子进程
      → stdin(JSON请求) → stdout(JSON响应)
      → Streamlit应用内推送
  immediate=True → 重大事件破例推送(score>0.85且时效<2h)
```

**选型理由**：
- 推送服务需随主进程启停，无独立生命周期
- 无需管理端口
- stdio部署复杂度最低
- 短连接、无状态推送

## 四、第三层：FC (Function Calling) — 逻辑简单、参数明确、紧耦合

FC工具是进程内直接调用的函数，无IPC开销，适合逻辑简单、参数明确、紧耦合的场景。

### 4.1 采集类FC（采集Agent内部调用）

| 工具名 | 依赖 | 特点 | 返回类型 |
|--------|------|------|---------|
| `fetch_rss` | feedparser | 并行采集，无状态 | `list[dict]` |
| `enrich_metadata` | LLM单次推理 | 分类/关键词提取 | `list[dict]` |
| `normalize_items` | 纯数据转换 | 无I/O | `list[dict]` |

### 4.2 排序类FC（排序Agent内部调用）

| 工具名 | 依赖 | 特点 | 返回类型 |
|--------|------|------|---------|
| `deduplicate` | ChromaDB SDK | ≥0.88/0.70-0.88 LLM裁决 | `list + 关系` |
| `rank_items` | 纯计算 | 多因子动态权重 | `list[dict]` |
| `vector_search` | ChromaDB SDK | 偏好向量检索 | `list[dict]` |
| `db_read` | SQLite | 反馈历史查询 | `list[dict]` |

### 4.3 简报类FC（简报Agent内部调用）

| 工具名 | 依赖 | 特点 | 返回类型 |
|--------|------|------|---------|
| `generate_briefing` | LLM | 合并摘要+简报生成 | `dict` |
| `reflect` | LLM | 质量评分+矛盾检查 | `dict` |

### 4.4 反馈类FC（反馈Agent内部调用）

| 工具名 | 依赖 | 特点 | 返回类型 |
|--------|------|------|---------|
| `update_preference` | EMA计算 | 偏好权重更新 | `dict` |
| `vector_add` | ChromaDB SDK | 偏好向量写入 | `list[str]` |

### 4.5 存储类FC（多个Agent共享调用）

| 工具名 | 依赖 | 特点 | 返回类型 |
|--------|------|------|---------|
| `db_write` | SQLite | 单次写入 | `bool` |
| `db_read` | SQLite | 无状态查询 | `list[dict]` |

## 五、选型原则（决策依据）

| 原则 | 判断条件 | 工具类型 | 示例 |
|------|---------|---------|------|
| **原则1** | 需独立部署或有状态 | **MCP** | search_web(外部HTTP), push_notification(需随主进程启停) |
| **原则2** | 纯函数逻辑或进程内SDK | **FC** | feedparser, ChromaDB SDK, SQLite, 纯计算 |
| **原则3** | 自主决策+多步推理 | **子Agent** | 采集需ReAct判断是否补充, 排序需ReAct判断是否调参 |
| **原则4** | MCP双传输选型 | SSE vs stdio | SSE→长连接+流式+需端口(search_web), stdio→短连接+子进程+无端口(push_notification) |

---

## 六、MCP双传输模式对比

| 对比维度 | SSE模式 (search_web) | stdio模式 (push_notification) |
|----------|----------------------|------------------------------|
| **传输协议** | HTTP SSE (长连接) | stdin/stdout pipe (短连接) |
| **端口管理** | 需管理 :8100 | 无端口 |
| **部署方式** | 独立进程常驻运行 | 子进程随主进程启停 |
| **生命周期** | 常驻运行 | 随主进程启停 |
| **适用场景** | 外部HTTP服务 + 连接池 | 内部推送 + 无状态服务 |
| **流式支持** | 支持(SSE stream) | 不支持 |
| **调用方式** | HTTP POST → SSE stream → 结果 | stdin(JSON) → stdout(JSON) → 结果 |
| **复杂度** | 需端口管理+连接池维护 | 部署复杂度最低 |
| **数据流** | 请求→SSE事件流→聚合结果 | 请求→单次响应→完成 |

## 八、调用链路汇总

### 主流程工具调用序列

```
understand_intent (FC: db_read → 偏好Goal)
  ↓
planner (FC: LLM Think → sub_agent_plan)
  ↓
invoke_sub_agent
  ├── 采集Agent (子图调用)
  │     ├── fetch_rss (FC)
  │     ├── search_web (MCP SSE :8100) ← 条件触发
  │     ├── enrich_metadata (FC)
  │     └── normalize_items (FC)
  ├── 排序Agent (子图调用)
  │     ├── vector_search (FC, ChromaDB)
  │     ├── deduplicate (FC)
  │     ├── rank_items (FC)
  │     └── db_read (FC, SQLite)
  ├── 简报Agent (子图调用)
  │     ├── generate_briefing (FC)
  │     └── reflect (FC)
  ↓
observe_results (FC: LLM Observe)
  ↓
reflect (FC: LLM质量审查)
  ↓
push_notification (MCP stdio → Streamlit推送)
  ↓
update_memory
  ├── db_write (FC, SQLite → execution_logs)
  ├── vector_add (FC, ChromaDB → 偏好向量更新)
```

### 反馈流程工具调用序列（独立）

```
用户反馈触发 (like/dislike/irrelevant)
  ↓
FeedbackAgent (子图调用, 异步)
  ├── update_preference (FC, EMA)
  ├── vector_add (FC, ChromaDB)
  ↓
偏好向量 → 影响后续排序Agent的 preference 因子
```

## 九、关键设计要点

1. **三种工具类型对应Prompt要求** — "skills(FC)、MCP、子Agent" 三类注册
2. **子Agent不是FC的包装** — 子Agent有独立StateGraph和自主决策能力（ReAct），是真正的Agent而非简单函数调用
3. **MCP只用2个** — MVP最小化外部依赖，search_web(SSE)和push_notification(stdio)各选最优传输模式
4. **FC进程内调用零开销** — 所有SQLite/ChromaDB/纯计算操作都用FC，避免不必要的IPC
5. **反馈Agent完全异步** — 不经过主Agent planner，直接由用户反馈触发
6. **子Agent可嵌套调用MCP和FC** — 采集Agent内部可以调用search_web(MCP)和fetch_rss(FC)，工具边界在Agent内而非Agent外

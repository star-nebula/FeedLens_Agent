# ADR-008: 简报输出格式 — 结构化 JSON + Markdown 渲染

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |
| **日期** | 2026-06-17 |
| **决策者** | 项目决策组（WB 独有决策） |
| **来源** | 决策 P6 |

## 上下文

LLM 生成的简报需要选择输出格式。简报是 FeedLens 的核心交付物，格式决定了后续处理能力（质量评分、渲染、存储）。

可选方案：
- **A: 结构化 JSON** — LLM 输出 JSON Schema 约束的结构化数据，再渲染为 Markdown
- **B: 直接 Markdown** — LLM 直接输出 Markdown 文本

## 决策

**采用方案 A**：结构化 JSON 输出 + Markdown 渲染。

## 理由

1. **项目描述要求**：「结构化简报（分类组织 + 重要性标注 + 来源引用）」天然对应 JSON Schema
2. **可校验**：JSON Schema 校验输出格式，LLM 输出格式错误可检测
3. **可自动评分**：brief_quality 评分需要结构化数据（completeness/relevance/coherence/score），Markdown 无法自动评分
4. **可触发重试**：score < 0.7 触发重试，依赖结构化的 quality 字段
5. **前端渲染解耦**：JSON 数据和渲染逻辑分离，更换前端框架不需要改 LLM 输出
6. **额外成本极低**：仅需在 LLM system prompt 中加一段 JSON Schema 约束

## 影响

- 简报 LLM 调用需要 JSON Schema 约束输出格式
- briefs 表同时存储 content_json 和 content_md
- generate_briefing 沿用 enrich_metadata 已提取的 importance/keywords，LLM 只做摘要组织
- quality 评分由代码层 `brief_quality_check_node` 独立计算（completeness×0.3 + relevance×0.4 + coherence×0.3），不再内嵌于 JSON 中
- 预检机制：`_preflight_for_briefing()` 生成前硬编码预检（过滤低质条目、URL去重、时间跨度检查）

## JSON Schema 关键字段（v2.2.0 平铺格式）

```json
{
  "title": "string (简报标题)",
  "summary": "string (简报摘要，200字以内)",
  "items": [
    {
      "id": "string (条目唯一标识)",
      "title": "string (条目标题)",
      "summary": "string (条目摘要)",
      "source": "string (来源名称)",
      "published_at": "string (发布时间)",
      "importance": "1-5 (重要性)",
      "url": "string (来源链接，必填)"
    }
  ],
  "generated_at": "string (ISO 时间戳)"
}
```

### v2.2.0 格式演进

- **移除 categories 分类逻辑**：旧版 items 按 category 分组、组内按 importance 降序；新版采用 items 平铺格式，每条条目独立呈现
- **条目结构完善**：URL 必填、主条目 ★ 标记、类似报道作为子条目展示
- **quality 评分解耦**：从 JSON 内嵌改为代码层独立计算，避免 LLM 自评的利益冲突
- **UI 兼容**：前端兼容新旧两种格式，无破坏性切换

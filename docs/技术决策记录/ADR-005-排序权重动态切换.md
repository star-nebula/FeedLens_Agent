# ADR-005: 排序权重动态切换 — 冷启动相似度优先 → 偏好优先

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |
| **日期** | 2026-06-17 |
| **决策者** | 项目决策组（WB 深度分析） |
| **来源** | 决策 T3、独特点 3/6/41 |

## 上下文

排序公式 `final_score = w₁·similarity + w₂·recency + w₃·preference + w₄·importance` 中，4 个因子的权重分配直接影响简报质量。核心矛盾：

- 冷启动阶段（用户无反馈历史）：偏好权重无数据支撑，逻辑上只能用相似度排序
- 有反馈后：偏好数据逐步积累，应该让偏好权重提升以体现个性化

可选方案：
- **A: 偏好优先（w_pref ≥ 0.30）始终** — Codex_DeepSeek/Codex_Mimo/GLM/Kimi
- **B: 相似度优先（w_sim ≥ 0.35）始终** — GPT/DeepSeek/Perplexity/TRAE/Qwen
- **C: 冷启动相似度优先 → 有反馈后动态切换偏好优先** — WB 提出

## 决策

**采用方案 C**：动态权重切换。

| 阶段 | 条件 | 权重配置 | 理由 |
|------|------|---------|------|
| 冷启动 | 用户反馈 < 3 条 | w₁=0.40, w₂=0.25, w₃=0.10, w₄=0.25 | 无偏好数据，相似度优先 |
| 有反馈 | 用户反馈 ≥ 3 条 | w₁=0.30, w₂=0.20, w₃=0.40, w₄=0.10 | 偏好数据充分，偏好优先 |

## 理由

1. **逻辑自洽**：冷启动阶段偏好权重没有数据来源，给 preference 设 0.30+ 等于用噪声排序。只有相似度和 importance 有实际数据
2. **项目描述佐证**：项目定位是「个性化筛选」，个性化只能在有反馈数据后生效
3. **借鉴 Codex_DeepSeek 的动态调整**：w₃ 从 0.10→0.40 的切换，是经过实验验证的合理跃迁幅度
4. **切换阈值 3 条**：3 条反馈足以建立初步偏好向量（like/dislike/irrelevant 各至少 1 条），切换后不会立即剧烈变化

## 补充决策

- **feedback_bias 与 preference 的关系**：feedback_bias 是偏好向量的即时补偿（用户刚给反馈但偏好向量尚未更新时的临时补偿），偏好向量 EMA 更新后 bias 自动归零。二者是时序互补，不是独立叠加
- **情节记忆不参与排序因子计算**：仅在冷启动阶段提供初始权重参考值

## 影响

- 排序模块需要维护两套权重配置
- 切换逻辑基于 feedback_count >= 3 条
- feedback_bias 作为临时补偿，偏好向量更新后自动清零

## 权重配置参数

| 参数 | 冷启动值 | 有反馈值 | 可调 |
|------|---------|---------|------|
| w_sim | 0.40 | 0.30 | 是 |
| w_recency | 0.25 | 0.20 | 是 |
| w_pref | 0.10 | 0.40 | 是 |
| w_importance | 0.25 | 0.10 | 是 |

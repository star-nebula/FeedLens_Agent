# ADR-012: 执行栅栏防并发 — per-user threading.Lock 串行化管线

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |
| **日期** | 2026-06-21 |
| **决策者** | 项目决策组 |
| **来源** | 架构演进 P2（可行性分析与MVP实施步骤.md）|

## 上下文

FeedLens 有三个触发源可能同时发生：
- APScheduler 的 cron 定时触发（每日简报）
- Streamlit UI 的手动触发（用户点「立即运行」）
- 重大事件破例推送（score > 0.85 且时效 < 2h 时即时触发）

三者都走唯一入口 `pipeline_runner.run_agent_pipeline`，最终在 `update_memory_node` 里对 ChromaDB `user_preference` 集合做偏好向量的「读-改-写」。如果两个管线并发执行，后写的会覆盖先写的偏好更新，导致用户反馈的个性化信号丢失——这是数据安全问题，不是性能问题。

## 决策

**per-user `threading.Lock`，单进程内串行化同一 user 的管线。**

- 新建 `utils/execution_fence.py`：`ExecutionFence` 类持有 `dict[user_id, Lock]`，`try_acquire_pipeline(user_id)` 非阻塞获取，成功返回锁对象、失败返回 None
- `run_agent_pipeline` 入口：获取锁失败则返回 `{"status": "skipped", "reason": "pipeline already running"}`；成功则 `try/finally` 保证 `lock.release()`
- 不同 user_id 互不影响，可并行执行

## 理由

**场景是「同进程内防止同 user 并发」，`threading.Lock` 就是最简且正确的解。**

| 备选方案 | 不选的原因 |
|---------|-----------|
| Redis 分布式锁 | MVP 是单进程 Streamlit + APScheduler，无跨进程场景，引入 Redis 是为不存在的需求付成本 |
| 文件锁 | 比线程锁重，且文件系统行为在 Windows/BaiduSyncdisk 下不可控 |
| 队列串行化 | 把并发触发改成排队执行，改变触发语义（破例推送应即时而非排队），且过度设计 |
| `RLock`（可重入锁） | 当前无重入场景（管线不会嵌套调用自己），普通 `Lock` 已够，`RLock` 多余 |

关键判断：
1. **锁粒度选 per-user 而非全局**：MVP 虽然单用户，但数据竞争只发生在同一 user 的偏好向量上，全局锁会无谓阻塞未来多用户场景
2. **非阻塞获取（`blocking=False`）**：定时触发撞上正在跑的管线时，应跳过本次而非排队等待——排队会导致下一次触发延迟、且堆积多个待执行任务
3. **`try/finally` 保证释放**：管线异常时锁也会释放，避免一次失败导致该 user 永久被锁死

## 影响

- 同一 user 的并发触发：一个正常执行，其余返回 `{"status": "skipped"}`，日志打印 `已有管线在执行，跳过本次触发`
- 不同 user 互不干扰，可并行（为未来多用户预留）
- 管线异常时锁正常释放，不会死锁

## 后续演进

**跨进程部署时失效，需替换为分布式锁。** 当前假设是单进程（ADR-010 部署方案确认 MVP 纯本地 Python 运行）。若未来拆成多进程（如 Streamlit 与 APScheduler 分进程），`threading.Lock` 将无法跨进程生效，届时需引入 Redis 分布式锁或基于数据库的乐观锁。文档中已标注此假设。

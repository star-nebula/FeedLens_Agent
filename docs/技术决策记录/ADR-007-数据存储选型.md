# ADR-007: 数据存储选型 — SQLite + ChromaDB

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |
| **日期** | 2026-06-17 |
| **决策者** | 项目决策组 |
| **来源** | 9/9 文档共识、决策 T9/I2 |

## 上下文

FeedLens 需要两类数据存储：
1. **结构化数据**：用户配置、RSS 源、采集条目、去重关系、简报、反馈、执行日志
2. **向量数据**：条目 embedding、偏好向量、语义记忆

可选方案（结构化）：
- SQLite — 轻量、零部署、单文件
- PostgreSQL — 生产级、功能丰富
- SQLAlchemy ORM — 透明映射

可选方案（向量）：
- ChromaDB — 轻量、本地运行、Python 原生
- Milvus — 分布式、高性能
- Pinecone — 云托管、零运维

## 决策

- **结构化数据 → SQLite（WAL 模式）**
- **向量数据 → ChromaDB**
- **ORM → 原生 SQL**
- **向量库交互 → FC 直接调用 ChromaDB SDK**

## 理由

**SQLite**：
1. 9/9 文档共识
2. 零部署、单文件，完美匹配 MVP 本地运行
3. WAL 模式支持读写并发
4. 原生 SQL 透明可调试（决策 I2）

**ChromaDB**：
1. 9/9 文档共识
2. 轻量、本地运行、Python 原生
3. 内置 embedding 函数和相似度搜索
4. 无需独立部署服务进程

**FC 直接调用 ChromaDB SDK**：
1. 进程内调用，无 IPC 开销
2. 开发成本更低（import chromadb vs 写 MCP Server 实现）
3. 延迟更低（函数调用 vs HTTP/RPC）

## 影响

- SQLite 11 张表 + ChromaDB 3 个 Collections
- ChromaDB 并发写入保护：feedback_workflow 与主流程异步运行时，对 user_preference collection 的写入通过串行化队列保证顺序
- 数据生命周期管理：raw_items 30天清理，ChromaDB 向量随 raw_items 同步删除

## 不采纳的方案

| 方案 | 不采纳理由 |
|------|----------|
| PostgreSQL | MVP 零部署需求；PostgreSQL 需安装和维护 |
| SQLAlchemy ORM | 原生 SQL 更透明可调试（决策 I2）；ORM 增加学习成本 |
| Milvus | MVP 数据量小，ChromaDB 够用；Milvus 需独立部署 |
| ChromaDB 作为 MCP Server | FC 直接调用无 IPC 开销，延迟更低（决策 T9） |

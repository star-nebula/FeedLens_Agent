# ADR-013: 模型回退链 — LLMRouter 顺序 try 防 LLM 不可用

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |
| **日期** | 2026-06-21 |
| **决策者** | 项目决策组 |
| **来源** | 架构演进 P4（可行性分析与MVP实施步骤.md）|

## 上下文

MVP 阶段 FeedLens 全链路依赖单个 LLM（DeepSeek Chat）。LLM 不可用时——API 限流、网络抖动、服务商宕机——planner 无法编排、采集元数据无法提取、简报无法生成，整条管线瘫痪。对信息聚合类 Agent，一次推送缺失可接受，但「一旦 LLM 出问题就完全无法产出」的脆弱性是结构性短板。

好在 `LLMProvider` 抽象基类（ABC）在 MVP 阶段就已定义，`DeepSeekProvider` 是一个实现。回退链只需在 ABC 之上加一层「多 Provider 顺序尝试」，调用方代码零改动——因为 Router 也是 `LLMProvider` 的子类。

## 决策

**`LLMRouter(list[LLMProvider])`：顺序 try 各 Provider，首个成功即返回，全失败才抛 RuntimeError。**

- `LLMRouter` 实现 `LLMProvider` ABC 的 `chat` / `chat_with_tools`，内部循环调用各 Provider
- `_get_llm_provider` 读 `config.llm.fallback`：有 `api_key` 则构建 `LLMRouter([primary, fallback])`，无则退化为仅用 primary（行为等同改动前）
- 配置激活：`config/config.yaml` 的 `fallback` 段填入备用 Provider 即启用（当前主 DeepSeek + 备 小米 mimo-v2.5）

## 理由

**ABC 已定义好合约，Router 就是「列表 + 循环 try」，不需要生产级特性。**

| 备选方案 | 不选的原因 |
|---------|-----------|
| 健康检查 + 自动标记不可用 | MVP 单进程、低频调用，主动探测的复杂度和成本超过收益。失败时直接 try 下一个，被动发现已足够 |
| 探测机制（先发心跳请求） | 增加每次调用的延迟和成本，MVP 无需 |
| 权重路由 / 负载均衡 | 回退链的目标是「兜底」而非「分流」，顺序 try 最简单且符合语义 |
| 异步并发请求取最快 | 成本翻倍且对 LLM 调用无意义（一次请求只要一个结果）|

关键判断：
1. **顺序 try 而非并行**：回退语义是「主不可用才用备」，不是「选最优」。顺序 try 成本最低、语义最清晰
2. **不引入健康检查**：被动发现（调用失败自动切下一个）在低频场景下足够。主动探测是为高并发生产场景设计的，FeedLens MVP 单用户每日几次调用，不值得
3. **保留退化路径**：`fallback.api_key` 为空时退化为单 Provider，确保未配置回退的环境行为不变，避免「不配置就报错」

## 影响

- 主 LLM 不可用时自动切换到备用 Provider，全链路不再因单点 LLM 故障瘫痪
- 备用 Provider 与主 Provider 输出风格差异可能影响 planner 决策质量，但「有回退」远优于「全链路降级」
- 未配置 fallback 时行为等同改动前，无破坏性
- 每次调用失败会打印 `[llm_router] {name} 调用失败，尝试下一个` 日志，便于排查

## 后续演进

**按需加生产级特性，当前不加。**
- 若 LLM 调用频率显著上升（多用户、高频触发），再加健康检查与自动标记不可用 Provider，避免每次都尝试一个已知挂掉的 Provider 浪费时间
- 若备用 Provider 数量增多（3+），再加优先级或权重，当前 2 个 Provider 顺序 try 已足够清晰
- 若观察到主备切换频繁，需评估是否主 Provider 稳定性本身有问题，而非靠回退链掩盖

# ADR-004: 向量去重策略 — 0.88 阈值 + 纯向量 + 模糊区间 LLM 裁决

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |

| **决策者** | 项目决策组（WB 深度分析） |

## 上下文

FeedLens 的核心功能之一是「智能去重」——同一事件不同来源的报道合并为一条。去重策略的三个维度需要决策：

1. **阈值选择**：高阈值（≥0.90）vs 中阈值（0.85-0.88）
2. **技术路线**：纯向量阈值判定 vs 多信号组合（NER/编辑距离）
3. **模糊区间处理**：一律判定 vs LLM 二元裁决

可选方案：
- **A: ≥0.90 纯向量一律判定** — 简单但可能误判跨语言同事件
- **B: 0.85-0.88 纯向量 + 模糊区间 LLM 裁决** — 精确但增加 LLM 调用
- **C: 0.88 纯向量 + 多信号（NER+编辑距离）** — 精确但复杂度高
- **D: 0.88 纯向量 + 模糊区间 LLM 裁决 + 校准脚本** — 精确、可调优、复杂度可控

## 决策

**采用方案 D**：0.88 初始阈值 + 纯向量 + 模糊区间（0.70-0.88）LLM 裁决 + 校准脚本。

- ≥0.88 → 判定为重复（高置信，无需 LLM）
- ≤0.70 → 判定为不重复（低置信，无需 LLM）
- 0.70-0.88 → 模糊区间，调用 LLM 做二元判断
- LLM 裁决上限 20 对（`max_llm_adjudications=20`），超限按 0.80 硬判
- 提供 `calibrate_dedup.py` 校准脚本，数据驱动优化阈值

## 理由

1. **0.88 是争议最小的阈值**：4 份文档用 0.88（Codex_DeepSeek/Codex_Mimo/Perplexity/Qwen），双方重叠最多
2. **纯向量去重开发成本最低**：不增加 NER/编辑距离等第二信号（决策 T2）
3. **模糊区间 LLM 裁决覆盖跨语言场景**：「苹果发布新 iPhone」vs「Apple 推出 iPhone 16」这类场景纯向量无法判断，但 LLM 二元裁决可以
4. **LLM 裁决调用量可控**：约 10-20% 的对进入模糊区间 ≈ 12-25 对（50 条采集），上限 20 对防止系统过慢
5. **校准脚本让阈值成为可优化的系统参数**：从硬编码变为数据驱动（标注 200 对 → P/R/F1 曲线 → 最优阈值）

## 影响

- 去重模块是排序前最耗时步骤（向量计算 + 可能的 LLM 裁决）
- `item_relations` 表记录去重关系和判定方法（vector_threshold / llm_adjudication）
- `calibrate_dedup.py` 是独立脚本，不影响主流程
- 裁决上限和硬判阈值作为配置参数（`max_llm_adjudications=20`, `dedup_hard_threshold=0.80`）

## 性能风险与缓解

| 风险 | 缓解策略 |
|------|---------|
| 50 条采集→C(50,2)=1225 对→10-20%模糊区间≈120-250次LLM调用 | max_llm_adjudications=20 上限；超限按 0.80 硬判 |
| LLM 裁决延迟 | 模糊区间裁决串行执行（StateGraph 保证），每次裁决 ~1s |
| 阈值偏离 0.88 | 校准脚本定期优化，配置参数可调 |

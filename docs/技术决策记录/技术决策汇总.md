# FeedLens 技术决策汇总

> **版本**：v1.0 MVP | **更新日期**：2026-06-20

---

## 技术选型总览

| 技术领域 | 选择方案 | 核心理由 |
|---------|---------|---------|
| Agent编排框架 | LangGraph StateGraph | 原生支持ReAct循环、子图、状态传递 |
| 系统架构 | 多Agent自主规划 | 核心差异化，Planner自主编排子Agent |
| MCP传输模式 | Push: stdio / Search: SSE | 本地推送无需端口，搜索支持流式返回 |
| 向量去重策略 | 0.88阈值 + 模糊区间LLM裁决 | 精确且可控，支持跨语言场景 |
| 排序权重 | 冷启动相似度优先 → 动态切换偏好优先 | 逻辑自洽，有反馈后才启用个性化 |
| Embedding模型 | 本地 bge-small-zh-v1.5 | 免费、无速率限制、中文效果好 |
| 数据存储 | SQLite + ChromaDB | 零部署、轻量、本地运行 |
| 简报输出格式 | 结构化JSON + Markdown渲染 | 可校验、可评分、可触发重试 |
| 用户反馈机制 | 三级反馈 + EMA偏好更新 | 区分"不相关"和"质量差"，平滑更新 |
| 部署方案 | MVP纯本地Python运行 | 迭代速度快，减少交付范围 |

## 详细决策表

| 技术 | 选择理由 | 不采纳方案 |
|------|---------|-----------|
| **Agent编排框架：LangGraph StateGraph** | 1. 9/9文档共识，零争议<br>2. 原生支持ReAct循环（planner→invoke→observe→再思考）<br>3. 原生支持子图，子Agent作为子图被主Agent调度<br>4. State TypedDict有状态传递机制<br>5. Python原生，与MVP技术栈无缝集成 | **CrewAI**：角色预设太强，不支持自定义ReAct循环<br>**AutoGen**：以对话为核心，不适合定时执行的pipeline-like Agent<br>**纯Python asyncio**：无编排框架支持，需手动实现状态管理、条件路由、重试逻辑 |
| **系统架构：多Agent自主规划** | 1. Prompt核心差异化要求，自主规划是核心卖点<br>2. Prompt明确要求子Agent作为工具注册类型<br>3. ReAct模式要求（思考→行动→观察→再思考）<br>4. 简历项目差异化，比cron+pipeline更有技术深度 | **单Agent+planner增强节点**：planner放在P1增强，MVP交付时没有展示核心价值<br>**单Agent+ReAct仅在关键节点**：只在collect_sources和rank_items有ReAct，不够自主 |
| **MCP传输模式：Push stdio / Search SSE** | **Push用stdio**：<br>1. 本地操作，无跨机器调用需求<br>2. 随主进程启停，无需端口管理<br>3. 5/8文档共识<br>**Search用SSE**：<br>1. 支持流式返回大搜索结果集<br>2. 独立进程管理连接池<br>3. 7/7文档共识<br>**实现框架**：FastMCP简化MCP Server开发，自动处理协议细节 | **Push用SSE**：SSE优势是跨机器远程调用，但MVP所有组件同机运行<br>**Search用stdio**：不支持流式返回，无法处理大搜索结果集<br>**原生MCP SDK**：需手动处理协议细节，开发成本更高 |
| **向量去重：0.88阈值 + 模糊区间LLM裁决** | 1. 0.88是争议最小的阈值，4份文档用0.88<br>2. 纯向量去重开发成本最低<br>3. 模糊区间LLM裁决覆盖跨语言场景<br>4. LLM裁决调用量可控（上限20对）<br>5. 校准脚本让阈值成为可优化参数 | **≥0.90纯向量一律判定**：简单但可能误判跨语言同事件<br>**0.85-0.88纯向量+多信号（NER+编辑距离）**：精确但复杂度高<br>**0.88纯向量+多信号**：开发成本高，MVP阶段不必要 |
| **排序权重：动态切换** | 1. 冷启动阶段偏好权重无数据支撑，只能用相似度排序<br>2. 项目定位是「个性化筛选」，个性化只能在有反馈数据后生效<br>3. 3条反馈足以建立初步偏好向量<br>4. 切换幅度（w_pref 0.10→0.40）经过实验验证 | **偏好优先始终（w_pref≥0.30）**：冷启动阶段偏好权重没有数据来源，等于用噪声排序<br>**相似度优先始终（w_sim≥0.35）**：无法体现个性化，与项目定位矛盾 |
| **Embedding模型：本地bge-small-zh-v1.5** | 1. 免费、无速率限制<br>2. 中文效果好（BAAI专门针对中文训练）<br>3. 推理速度快（~50ms/条）<br>4. 开发调试体验好（本地模型一次下载、无限调用）<br>5. 预留API切换接口 | **API text-embedding-v3**：有速率限制和成本，开发调试阶段频繁重跑不计费是关键需求<br>**bge-large-zh**：参数量更大（326M vs 153M），推理速度慢2-3倍，MVP阶段小模型够用 |
| **数据存储：SQLite + ChromaDB** | **SQLite**：<br>1. 9/9文档共识<br>2. 零部署、单文件，完美匹配MVP本地运行<br>3. WAL模式支持读写并发<br>**ChromaDB**：<br>1. 9/9文档共识<br>2. 轻量、本地运行、Python原生<br>3. 内置embedding函数和相似度搜索 | **PostgreSQL**：生产级但MVP不需要<br>**Milvus**：分布式高性能但MVP不需要<br>**Pinecone**：云托管但MVP本地运行<br>**SQLAlchemy ORM**：原生SQL更透明可调试 |
| **简报输出：结构化JSON + Markdown渲染** | 1. 项目描述要求「结构化简报」天然对应JSON Schema<br>2. 可校验、可自动评分、可触发重试<br>3. 前端渲染解耦，更换前端框架不需要改LLM输出<br>4. 额外成本极低（仅需在prompt中加JSON Schema约束） | **直接Markdown**：无法自动评分，无法触发重试，无法校验输出格式 |
| **用户反馈：三级反馈 + EMA偏好更新** | **三级反馈**：<br>1. 区分「不相关」和「质量差」语义不同<br>2. irrelevant信号让系统从候选集移除整类内容<br>**EMA平滑**：<br>1. 避免偏好剧烈波动<br>2. 偏好变化渐进式，符合人类偏好变化规律<br>**正负分离**：<br>1. 区分「喜欢什么」和「不喜欢什么」<br>2. 偏好清理更精准 | **二元反馈（like/dislike）**：无法区分「不相关」和「质量差」<br>**四级反馈（+valuable）**：valuable与like语义重叠<br>**简单累加**：连续3次dislike可能使偏好突变<br>**单一偏好向量**：无法区分正负偏好 |
| **部署方案：MVP纯本地Python运行** | 1. 用户明确决定「纯本地不打包」<br>2. Prompt约束「本地Python直接运行」<br>3. MVP速度优先，频繁改代码、重启服务、看日志<br>4. 减少交付范围，聚焦核心功能 | **MVP本地 + Docker Compose交付**：增加交付物，MVP阶段不必要<br>**MVP纯本地 + docker-compose.yml备用**：不作为交付物，只写参考配置，增加维护成本 |

## 技术栈速查表

| 组件 | 选型 | 版本/配置 | 说明 |
|------|------|----------|------|
| Agent编排 | LangGraph StateGraph | - | 主Agent + 5个子Agent |
| LLM | DeepSeek Chat | - | 通过LLMProvider抽象 |
| Embedding | bge-small-zh-v1.5 | 153M参数 | 本地离线推理 |
| 向量数据库 | ChromaDB | 3个Collections | feed_items / user_preference / domain_knowledge |
| 关系数据库 | SQLite | WAL模式，11张表 | 零部署、单文件 |
| 定时任务 | APScheduler | cron触发 | 每日简报定时生成 |
| MCP框架 | FastMCP | - | 简化MCP Server开发 |
| MCP搜索服务 | SSE | :8100 | 流式返回搜索结果 |
| MCP推送服务 | stdio | - | 本地通知队列 |
| HTTP客户端 | requests + httpx | - | 连接池 + 重试 + 异步 |
| RSS解析 | feedparser | - | 标准RSS/Atom解析 |
| 并发模型 | ThreadPoolExecutor | max_workers=10 | RSS并行采集 |
| 用户界面 | Streamlit | - | 多页面应用 |
| 日志 | structlog | - | 结构化日志 |

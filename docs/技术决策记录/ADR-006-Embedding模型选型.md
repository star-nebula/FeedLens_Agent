# ADR-006: Embedding 模型选型 — 本地 bge-small-zh-v1.5

| 元数据 | |
|--------|--------|
| **状态** | 已采纳 |
| **日期** | 2026-06-17 |
| **决策者** | 项目决策组 |
| **来源** | WB+TRAE 共识、决策 T4 |

## 上下文

FeedLens 的核心操作依赖 Embedding：
- 去重：计算条目间 cosine 相似度
- 排序：偏好向量与条目向量的相似度计算
- 记忆检索：ChromaDB 向量相似度搜索

可选方案：
- **A: 本地 sentence-transformers（bge-small-zh-v1.5）** — 5份文档
- **B: API 调用（LLM text-embedding-v3）** — 2份文档

## 决策

**采用方案 A**：本地 bge-small-zh-v1.5。

## 理由

1. **免费、无速率限制**：MVP 开发阶段频繁重跑测试，API 调用会累积成本且受速率限制
2. **中文效果好**：bge-small-zh-v1.5 是 BAAI 专门针对中文训练的模型，153M 参数
3. **推理速度快**：~50ms/条，10 条条目的向量计算 < 1s
4. **开发调试体验好**：本地模型一次下载、无限调用，不需要管理 API Key 和调用配额
5. **预留 API 切换接口**：LLMProvider 抽象类预留了 embedding provider 扩展点，后续可无痛切换

## 影响

- 首次运行需下载模型约 100MB，应显示进度条
- 所有向量操作使用统一的 Embedding 模型（去重/排序/记忆检索）
- ChromaDB Collections 存储的向量与实时计算使用同一模型，确保一致性
- 后续如果需要多语言支持，可切换为 BGE-M3（P2）

## 不采纳的方案

| 方案 | 不采纳理由 |
|------|----------|
| API text-embedding-v3 | 有速率限制和成本；开发调试阶段频繁重跑不计费是关键需求 |
| bge-large-zh | 参数量更大（326M vs 153M），推理速度慢 2-3 倍；MVP 阶段小模型够用 |
